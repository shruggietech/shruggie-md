## Plan Overview

This document defines the full build sequence for the Shruggie Markdown MVP (version 0.1.0). It is organized into 10 sprints ordered by dependency chain. Each sprint is scoped for delegation to one or more sub-agent processes. Sprints are sequential at the top level, but tasks within a sprint may be parallelized where noted.

The authoritative source for all implementation details is `shruggie-md-spec.md`. This plan references spec sections by number (e.g., §4.3) rather than duplicating their content. When a conflict exists between this plan and the spec, the spec wins.

### Dependency Graph (Simplified)

```
Sprint 1: Foundation
    └─► Sprint 2: Design System & UI Primitives
         └─► Sprint 3: Markdown Rendering Pipeline
              └─► Sprint 4: Application Shell & Full-View
                   ├─► Sprint 5: Editor & Split-View
                   │    └─► Sprint 7: File Ops & Conversion
                   ├─► Sprint 6: Configuration & Settings
                   │    └─► Sprint 8: Library Mode
                   └─► Sprint 9: Chrome Extension & PWA
                        └─► Sprint 10: Desktop Distribution & QA
```

### Technology Stack Summary

Refer to §3.1 of the spec for the full stack table. Key decisions that affect every sprint:

- Desktop shell: Tauri 2 (Rust backend)
- Frontend: React 18+ with TypeScript
- Styling: Tailwind CSS 4 with custom design tokens as CSS custom properties
- Editor: CodeMirror 6
- Default markdown engine: markdown-it
- Build tooling: Vite
- Icons: Lucide

## Sprint 1 — Foundation & Project Scaffolding

**Goal:** Establish the monorepo structure, build tooling, and platform abstraction skeleton so that all subsequent sprints have a working development environment.

### 1.1 Repository Initialization

- Create the repository structure defined in §12. Every directory listed must exist, even if initially populated with only an `index.ts` barrel export.
- Initialize `package.json` with the project slug `shruggie-md`.
- Initialize `tsconfig.json` with strict mode enabled, path aliases for `@/components`, `@/engines`, `@/linters`, `@/platform`, `@/hooks`, `@/types`, `@/styles`.

### 1.2 Vite Configuration

- Configure Vite as the build tool (§3.1).
- Set up three build targets in `vite.config.ts`:
  - Default web build (used by PWA).
  - Tauri integration build (via `@tauri-apps/vite-plugin` or equivalent).
  - Chrome extension build (outputs to `extension/dist/`).
- Confirm hot module replacement works for the default web target.

### 1.3 Tauri Initialization

- Scaffold the `src-tauri/` directory with `Cargo.toml`, `tauri.conf.json`, and `src/main.rs`.
- Configure `tauri.conf.json` with the product name "Shruggie Markdown", identifier `com.shruggietech.shruggie-md`, and window defaults (dark title bar where supported).
- Verify the Tauri dev server launches and displays a blank React root.

### 1.4 Tailwind CSS 4 Setup

- Install Tailwind CSS 4 and configure it in `tailwind.config.ts`.
- Extend the Tailwind theme with the full set of design tokens from §4.3 through §4.5 (colors, spacing, radii, typography). These tokens must be defined as CSS custom properties at `:root` and consumed via Tailwind's configuration, not hardcoded into component classes.
- Create `src/styles/tokens.css` as the single source of truth for all CSS custom properties.

### 1.5 Platform Abstraction Layer

- Create `src/platform/platform.ts` as described in §3.3. This module exports a uniform API covering:
  - `readFile(path): Promise<string>`
  - `writeFile(path, content): Promise<void>`
  - `watchFile(path, callback): Unsubscribe`
  - `openFileDialog(extensions): Promise<string | null>`
  - `openDirectoryDialog(): Promise<string | null>`
  - `getStoragePath(): string`
  - `readConfig(): Promise<Config>`
  - `writeConfig(config): Promise<void>`
  - `getPlatformCapabilities(): PlatformCapabilities`
- The `PlatformCapabilities` type must include boolean flags: `hasFilesystem`, `hasFileWatcher`, `hasNativeDialogs`, `hasCliArgs`.
- Implement the Tauri adapter (`src/platform/tauri.ts`) as the first concrete implementation. Stub the Chrome extension and PWA adapters with no-op implementations that return sensible defaults.
- Platform detection at startup: query the runtime environment and bind the correct adapter. The frontend must never import platform adapters directly; all access goes through `platform.ts`.

### 1.6 TypeScript Type Definitions

- Create `src/types/config.ts` with the full `Config` interface covering every setting in §10 (appearance, editor, preview, engine, file extensions, library).
- Create `src/types/platform.ts` with the `PlatformCapabilities` interface.
- Create `src/types/engine.ts` with the `MarkdownEngine` interface (input: string, output: string).
- Create `src/types/linter.ts` with the `LinterAdapter` interface (input: string, output: array of diagnostics).
- All compiled defaults for configuration (§10.2 through §10.4) must be defined in a `src/config/defaults.ts` file.

### Acceptance Criteria

- `npm run dev` launches the Vite dev server and renders an empty React root.
- `npm run tauri dev` launches a Tauri window displaying the same React root.
- All CSS custom properties from §4.3 through §4.5 are defined and available.
- The platform abstraction resolves to the Tauri adapter when running under Tauri and falls back to a no-op web adapter otherwise.
- TypeScript compilation succeeds with zero errors and strict mode enabled.

## Sprint 2 — Design System & UI Primitives

**Goal:** Implement the full visual design language (§4) and the shared component library (`src/components/common/`) so that all subsequent UI work uses a consistent, spec-compliant foundation.

### 2.1 Theme System

- Implement the `data-theme` attribute scoping system described in §4.3.
- Define three theme modes: `light`, `dark`, `system`. Default to `dark`.
- The `system` mode must track `prefers-color-scheme` via a `matchMedia` listener and update the `data-theme` attribute reactively.
- Implement theme switching with the 250ms CSS transition on `[data-theme]` as specified in §4.7.
- Implement `prefers-reduced-motion` detection: when active, all transition durations must be set to `0ms` (§4.7).

### 2.2 Visual Styles

- Implement the named visual style system described in §10.2. Styles are layered CSS custom property overrides on top of the base light/dark tokens.
- Create four initial styles: `default`, `warm`, `cool`, `monochrome`.
- Each style is a separate CSS file (or JS object) that remaps a subset of the token values from §4.3.
- Create a `useTheme` hook (`src/hooks/useTheme.ts`) that manages the active color mode and visual style, reads from and writes to configuration state.

### 2.3 Shared UI Components

Build the following components in `src/components/common/`. All components must consume design tokens exclusively (no raw hex, no magic numbers). Reference §4.6 for every dimension, color, and interaction detail.

**Button**
- Two variants: `ghost` (default) and `accent`.
- Props: `variant`, `icon` (Lucide icon component), `tooltip` (string), `onClick`, `disabled`.
- Minimum tap target: 32px. Focus outline: 2px offset, accent color.
- Hover transitions: 120ms ease-out (§4.7).

**Input**
- Text input with background, border, and focus behavior per §4.6.
- Props: `value`, `onChange`, `placeholder`, `type`.

**Modal**
- Centered overlay with backdrop (`rgba(0,0,0,0.5)`).
- Entrance animation: fade + scale from 0.97 over 150ms with the specified cubic-bezier (§4.7).
- Props: `isOpen`, `onClose`, `title`, `children`.
- Max width: 480px. Content padding: `--space-6`.

**Tooltip**
- Appears on 400ms hover delay, fades in over 120ms (§4.7).
- Props: `content` (string), `children` (trigger element).
- Styling per §4.6.

**Divider (Draggable)**
- Vertical pane divider for split-view. Default: 1px solid border-primary.
- On hover: thickens to 3px, color shifts to accent-subtle.
- Drag behavior: immediate (0ms), no transition during active drag (§4.7).
- Emits `onResize(leftWidthPercent)` during drag.
- Enforces minimum pane width of 280px (§5.3).

### 2.4 Iconography

- Install `lucide-react`.
- Create a thin wrapper (`src/components/common/Icon.tsx`) that standardizes icon rendering at 16px with 1.5px stroke weight and `currentColor` inheritance (§4.8).

### Acceptance Criteria

- Toggling between light, dark, and system themes updates all surfaces, text, borders, and accents correctly. The 250ms transition is visible (and suppressed under `prefers-reduced-motion`).
- All four visual styles produce visibly distinct but cohesive palettes.
- Each shared component renders correctly in both light and dark mode with correct sizing, padding, transitions, and interaction states.
- Draggable divider resizes two placeholder panes smoothly with the 280px minimum enforced.

## Sprint 3 — Markdown Rendering Pipeline

**Goal:** Implement the engine abstraction layer (§8), all three selectable engines, the sanitization pipeline, and the preview content stylesheet so that raw markdown can be compiled and displayed.

### 3.1 Engine Abstraction

- Create `src/engines/index.ts` exporting a `renderMarkdown(source: string, engineId: string): string` function.
- The function resolves the requested engine, compiles the source, and returns raw HTML. Engine selection is driven by the configuration value for active engine (§8.2).
- Define the `MarkdownEngine` interface: `{ id: string; name: string; compile(source: string): string }`.

### 3.2 Engine Implementations

Implement three adapters in `src/engines/`:

**markdown-it** (`markdown-it-engine.ts`)
- Default engine. Install `markdown-it` and configure with GFM tables, task lists, and footnotes via `markdown-it-footnote`, `markdown-it-task-lists`, and the built-in table support.
- Enable linkify and typographer options.

**marked** (`marked-engine.ts`)
- Install `marked`. Configure with GFM mode enabled.

**remark** (`remark-engine.ts`)
- Install `remark`, `remark-html`, `remark-gfm`.
- Pipeline: `remark().use(remarkGfm).use(remarkHtml).processSync(source)`.

All three must produce clean HTML strings suitable for the sanitization step.

### 3.3 Sanitization

- Install `dompurify`.
- Create `src/engines/sanitize.ts` wrapping DOMPurify with a configuration that allows standard markdown-generated HTML elements (headings, paragraphs, lists, tables, code blocks, images, links, blockquotes, hr, task list checkboxes) while stripping script tags, event handlers, and other XSS vectors.
- This sanitization step runs after every engine compilation, unconditionally (§8.1).

### 3.4 Code Block Highlighting

- Install `highlight.js` (or Shiki; spec says configurable, default to highlight.js for MVP simplicity).
- Create a post-render hook (`src/engines/highlight.ts`) that finds all `<pre><code>` elements in the rendered output and applies syntax highlighting.
- The hook runs after sanitized HTML is injected into the DOM.

### 3.5 Preview Stylesheet

- Create `src/styles/preview.css` implementing every rendered content style from §8.3: headings (h1-h6 with descending sizes and bottom borders on h1/h2), paragraphs, code blocks, inline code, blockquotes, tables, links, images, horizontal rules, and task lists.
- All styles reference CSS custom properties. No raw values.
- Maximum reading width: 720px, centered, with `--space-8` horizontal padding (§5.2).

### Acceptance Criteria

- A hardcoded markdown string rendered through each of the three engines produces visually identical (or near-identical) output in the preview area.
- Markdown containing `<script>` tags, `onerror` attributes, or other XSS payloads renders safely with malicious content stripped.
- Fenced code blocks with language annotations display correct syntax highlighting.
- All rendered element styles match §8.3 in both light and dark mode.

## Sprint 4 — Application Shell & Full-View

**Goal:** Build the toolbar (§4.6, §5.1), view mode routing, keyboard shortcut system, and the full-view layout so that a user can open the application and see rendered markdown.

### 4.1 Toolbar

- Implement `src/components/Toolbar/Toolbar.tsx` per §4.6: single horizontal bar, 40px height, `--color-bg-secondary` background, bottom border.
- Toolbar contents (left to right):
  - Application name or logo area (minimal).
  - View mode toggle buttons (Full-view, Split-view, Library, Settings) using ghost-variant icon buttons with tooltips.
  - File name display (shows the name of the currently open file, or "No file" / URL when applicable).
  - Action buttons area (placeholder slots for conversion and remote-fetch actions, implemented in later sprints).
  - Status area (placeholder for lint summary count, implemented in Sprint 5).
- Library button visibility is conditional on `PlatformCapabilities.hasFilesystem` (§3.3).

### 4.2 View Mode State Management

- Create a `useViewMode` hook managing the active view: `full-view`, `split-view`, `library`, `settings`.
- Default view on launch: `full-view` (§5.2).
- View transitions use the 200ms ease-in-out animation on width and opacity (§4.7).

### 4.3 Keyboard Shortcuts

- Implement the four shortcuts from §5.1:
  - `Ctrl/Cmd+1` — Full-view
  - `Ctrl/Cmd+2` — Split-view
  - `Ctrl/Cmd+3` — Library (desktop only)
  - `Ctrl/Cmd+,` — Settings
- Use a centralized keyboard handler that respects platform modifier keys (Cmd on macOS, Ctrl elsewhere).

### 4.4 Full-View Layout

- Implement `src/components/Preview/Preview.tsx` as the full-view rendering surface.
- Content area: horizontally centered, max-width 720px, horizontal padding `--space-8` on each side (§5.2).
- Wire the rendering pipeline from Sprint 3: accept a markdown source string, compile through the active engine, sanitize, inject into the preview container, run the highlight post-render hook.

### 4.5 File Open (Desktop)

- Wire the Tauri platform adapter's `openFileDialog` to a toolbar button or `Ctrl/Cmd+O`.
- Filter the dialog by the configured file extension whitelist (§10.4). Use compiled defaults until the config system is built in Sprint 6.
- On file selection: read the file via the platform adapter, pass the content to the preview component.
- Reject files with non-matching extensions with a user-visible error message suggesting the user modify the extension whitelist in settings (§10.4).

### Acceptance Criteria

- The application launches with an empty full-view and a functional toolbar.
- Pressing `Ctrl/Cmd+O` on desktop opens a file dialog filtered to markdown extensions. Selecting a `.md` file renders it in the preview pane with correct styling.
- View mode toggle buttons switch between placeholder views (full-view renders content; split-view, library, and settings show labeled placeholders).
- All four keyboard shortcuts work.

## Sprint 5 — Editor & Split-View

**Goal:** Integrate CodeMirror 6 (§7), implement split-view mode (§5.3) with scroll synchronization, and wire up the linting system (§7.3).

### 5.1 CodeMirror Integration

- Implement `src/components/Editor/Editor.tsx` wrapping a CodeMirror 6 instance.
- Install and configure all required extensions from §7.1: `@codemirror/lang-markdown`, `@codemirror/language-data`, `@codemirror/search`, `@codemirror/autocomplete`, `@codemirror/lint`, `@codemirror/view`, `@codemirror/state`.
- Build a custom CodeMirror theme at runtime from the active CSS custom property values (§7.1). The editor theme must update reactively when the application theme changes.

### 5.2 Syntax Highlighting

- Configure markdown syntax highlighting per §7.2: headings (bold, scaled size), bold/italic markers (dimmed), links (URL secondary, text accent), code spans (monospace + hover-bg), blockquotes (left border accent), list markers (accent color), horizontal rules.
- Fenced code blocks must receive language-specific highlighting via `@codemirror/language-data` (§7.2).

### 5.3 Split-View Layout

- Implement the split-view container that places the Editor on the left and Preview on the right (§5.3).
- Default split ratio: 50/50.
- Use the Draggable Divider component from Sprint 2 to allow pane resizing.
- Minimum pane width: 280px (§5.3).
- Persist the split ratio to configuration (once the config system exists in Sprint 6; for now, store in React state).

### 5.4 Scroll Synchronization

- Implement proportional scroll sync as described in §5.3: editor scroll percentage maps to preview scroll percentage.
- Do not use per-line tracking. The proportional approach avoids jitter from content that expands during rendering (tables, images, code blocks).
- Sync direction: editor cursor movement drives preview scroll position.

### 5.5 Live Preview

- Wire editor content changes to the rendering pipeline with a 150ms debounce (§6.2).
- On every debounced change: recompile through the active engine, sanitize, update the preview container, re-run the highlight hook.

### 5.6 Linting

- Implement `src/linters/markdownlint-adapter.ts` adapting `markdownlint` output to the CodeMirror `Diagnostic` interface.
- Implement `src/linters/remark-lint-adapter.ts` adapting `remark-lint` output to the same interface.
- Create a custom CodeMirror lint source function that calls the active linter adapter and returns diagnostics (§7.3).
- Lint diagnostics display: wavy underlines for errors, dotted for warnings. Tooltip on hover with rule ID, severity, and description (§7.3).
- Lint summary count (errors/warnings) displayed in the toolbar status area (§7.3).
- Linting is disabled by default (§10.3). The toggle will be wired in Sprint 6 via settings.

### Acceptance Criteria

- Split-view displays the editor and preview side by side with correct syntax highlighting in the editor.
- Typing in the editor updates the preview within 150ms (debounced).
- Dragging the divider resizes panes smoothly with the 280px minimum enforced.
- Scroll sync works: scrolling the editor causes the preview to scroll proportionally.
- Enabling linting (via a temporary flag) shows diagnostics inline and in the toolbar.
- Switching linters produces different diagnostic output for the same source.

## Sprint 6 — Configuration & Settings UI

**Goal:** Build the configuration persistence system (§10.1), all three platform storage adapters, and the full settings panel UI (§5.5) so that every user preference is functional.

### 6.1 Configuration Architecture

- Implement `src/config/store.ts` as the central configuration state manager.
- On startup: load persisted config via the platform adapter, merge with compiled defaults from `src/config/defaults.ts`. Unknown keys are silently ignored (§10.1).
- On change: write the full config object back to platform storage.
- Expose configuration via a `useConfig` hook that provides read access and setter functions for individual config keys.

### 6.2 Platform Storage Adapters

- **Tauri** (`src/platform/tauri.ts`): Read/write JSON to the platform-standard path (§10.5). Windows: `%LOCALAPPDATA%\shruggie-md\config.json`. Linux: `$XDG_CONFIG_HOME/shruggie-md/config.json`. macOS: `~/Library/Application Support/shruggie-md/config.json`.
- **Chrome extension** (`src/platform/chrome.ts`): Use `chrome.storage.sync` API. Serialize config as a single JSON string under one key.
- **PWA** (`src/platform/pwa.ts`): Use IndexedDB with a store named `shruggie-md-config` (§10.5). Use a thin wrapper (e.g., `idb-keyval` or a manual implementation).

### 6.3 Settings UI

- Implement `src/components/Settings/Settings.tsx` as a single scrollable panel with card-like sections (§5.5). Visual treatment: Apple System Settings / Linear Settings pattern, clean vertical stacking, generous whitespace, no tabs.
- Implement all six sections in order (§5.5):

**Appearance** — Color mode toggle (Light/Dark/System), visual style dropdown (default, warm, cool, monochrome). Wire to the theme system from Sprint 2.

**Editor** — Font family dropdown (curated list of monospace stacks, plus custom input), font size number input, line height number input, show-line-numbers toggle, word-wrap toggle, linting-enabled toggle, active-linter dropdown (markdownlint / remark-lint). All defaults from §10.3.

**Preview** — Font family dropdown (curated list including serif options, plus custom input), font size, line height. Wire to the preview stylesheet.

**Markdown Engine** — Engine dropdown (markdown-it, marked, remark). Wire to the engine abstraction from Sprint 3.

**File Extensions** — Extension whitelist editor. Display the current set (default: `.md`, `.markdown`, `.mdown`, `.mkdn`, `.mkd` per §10.4). Allow adding and removing extensions. Input validation: must start with a dot, alphanumeric characters only after the dot.

**Library** (desktop only, hidden when `hasFilesystem` is false) — Mount directory button (triggers directory picker dialog), display of currently mounted path, recursion toggle (default: on), hidden-file toggle (default: hidden excluded), independent-extension-rules toggle. When the independent toggle is on, display a second extension whitelist editor specific to library mode (§9.5).

### 6.4 Wire All Settings to Live Behavior

- Every setting must take effect immediately. No restart required.
- Theme and style changes: update `data-theme` and style class on `document.documentElement`.
- Editor font/size/line-height changes: reconfigure the CodeMirror instance.
- Preview font/size/line-height changes: update CSS custom properties.
- Engine change: recompile the current document through the newly selected engine.
- Linter change: re-lint the current document through the newly selected linter.

### Acceptance Criteria

- Settings panel renders all six sections with correct visual treatment.
- Changing any setting persists across application restarts (test on Tauri).
- Theme, engine, linter, and font changes apply immediately without reload.
- The file extension editor correctly adds and removes extensions, and the modified set is used by the file-open dialog.
- Library settings section is hidden in non-desktop contexts.

## Sprint 7 — File Operations & Conversion

**Goal:** Implement file saving (§6.2), live reload/file watching (§6.1), markdown-to-HTML export (§6.3), markdown-to-PDF export (§6.3), and remote URL fetching (§6.4).

### 7.1 File Saving

- **Desktop (Tauri):** `Ctrl/Cmd+S` writes editor content to the source file atomically (write to temp file, then rename) as specified in §6.2.
- **Chrome extension and PWA:** `Ctrl/Cmd+S` triggers a file download to the user's default download location (§6.2).
- Save is only available when the source is a local file (not remote content). Disable the save action for remote content (§6.4).

### 7.2 Live Reload (Desktop Only)

- Implement the file watcher using Tauri's filesystem watch API (§6.1).
- When the open file's mtime changes, re-read the file and re-render the preview.
- Debounce: 300ms (§6.1).
- Visual indicator: a brief 200ms pulse of `--color-accent-subtle` on the preview background when content refreshes (§5.2).
- The watcher must be torn down and re-created when the user opens a different file.

### 7.3 Markdown-to-HTML Export

- `Ctrl/Cmd+Shift+H` or toolbar button (§6.3).
- Generate a self-contained HTML file with all styles inlined. No external stylesheet references.
- The inlined styles must include the preview stylesheet (§8.3) and the active theme's color token values, producing a standalone document that looks identical to the in-app preview.
- Trigger a file-save dialog (desktop) or download (browser targets).

### 7.4 Markdown-to-PDF Export

- `Ctrl/Cmd+Shift+P` or toolbar button (§6.3).
- Pipeline:
  1. Compile markdown to HTML.
  2. Apply the print-specific stylesheet (`src/styles/print.css`).
  3. Replace `<hr>` elements with `page-break-before: always` (§6.3).
  4. Invoke the browser print dialog (or Tauri's direct PDF write if available).
- Print stylesheet requirements: standard print margins, page numbers in footer, no URL footers, clean page breaks at `<hr>` positions.

### 7.5 Remote URL Fetching

- Add a URL input field to the toolbar (§6.1, §6.4).
- On submit: send a plain HTTP `GET` request (no auth headers).
- Validate the response content-type is a text type. Display an error if not (§6.1).
- Render fetched content in full-view. The editor is available in split-view for inspection, but save is disabled (§6.4).
- Fetched content is not cached to disk (§6.4).
- Add a small info badge (`ⓘ`) next to the URL input that opens a modal with usage instructions for remote fetching (§6.1).
- CLI support on desktop: `shruggie-md --url <url>` (§6.1).
- CORS: browser targets are subject to CORS restrictions. The Chrome extension uses `host_permissions` to bypass this (handled in Sprint 9).

### Acceptance Criteria

- Editing in split-view and pressing `Ctrl/Cmd+S` on desktop writes the file. Reopening the file shows the saved content.
- Modifying a file externally while it is open in Shruggie triggers an automatic re-render with the accent-subtle flash.
- HTML export produces a standalone file that renders correctly when opened in a browser.
- PDF export opens the print dialog with correct page breaks at `<hr>` positions and page numbers in the footer.
- Pasting a public raw markdown URL (e.g., a raw GitHub link) into the URL field fetches and renders the content. Save is visibly disabled.

## Sprint 8 — Library Mode

**Goal:** Implement the full library feature set (§9) for the desktop target: directory mounting, recursive traversal, the sortable/filterable table, and library-specific extension rules.

### 8.1 Directory Mounting

- Wire the "Mount directory" button in Library settings (Sprint 6) to the platform adapter's `openDirectoryDialog`.
- Store the mounted path in configuration.
- Only one directory may be mounted at a time. Changing the mount replaces the library entirely (§9.2).

### 8.2 File Scanning

- On mount (and on refresh), scan the mounted directory for files matching the active extension rules.
- Default behavior: recursive traversal into all subdirectories (§9.4).
- Respect the recursion toggle: when off, only top-level files.
- Respect the hidden-file toggle: when on (default), exclude files and directories whose names start with `.` (§9.4).
- Non-matching files are silently ignored (§9.4).
- For each matching file, collect: filename (without extension), absolute path, `mtime` (ISO 8601 with UTC offset), `birthtime` (ISO 8601 with UTC offset).

### 8.3 Library Table

- Implement `src/components/Library/Library.tsx` displaying the sortable data table described in §9.3.
- Four columns: Title, Path, Last Edited, Created.
- Sort behavior: click a column header to sort by that column. Shift-click to add a secondary sort (§9.3).
- Default sort: Last Edited, newest first.
- Text filter: a search input in the toolbar that performs substring matching against Title and Path columns (§9.3).
- Clicking a row opens the file in full-view mode (§9.3).

### 8.4 Extension Rules Override

- When the independent-extension-rules toggle in Library settings is off, library mode uses the global extension whitelist (§9.5).
- When the toggle is on, library mode uses its own independent extension list, editable within the Library settings section (§9.5).
- Changing the toggle or modifying extensions triggers an immediate rescan.

### 8.5 Toolbar Adaptation

- When the library view is active, the toolbar adapts to show library-specific controls: mount directory button, refresh button, and the filter input (§5.4).
- File-specific controls (save, export, URL input) are hidden in library view.

### Acceptance Criteria

- Mounting a directory populates the library table with all markdown files found recursively.
- Disabling recursion reduces the table to top-level files only.
- Enabling hidden-file inclusion surfaces files inside `.git` and similar directories.
- Sorting by each column works correctly, including multi-column sorting via shift-click.
- The text filter narrows the table to matching rows in real time.
- Clicking a row opens the file in full-view with correct rendering.
- Toggling independent extension rules and modifying the library-specific extension list triggers a rescan.

## Sprint 9 — Chrome Extension & PWA

**Goal:** Build the two browser-based distribution targets: the Chrome extension (§11.2) and the Progressive Web App (§11.3).

These two targets can be built in parallel by separate sub-agents since they share the frontend core but have independent shell code.

### 9.1 Chrome Extension

**Manifest V3 Setup**
- Create `extension/manifest.json` per §11.2: Manifest V3, `content_scripts` with match patterns for recognized markdown extensions, `host_permissions` for CORS bypass, `action` for the browser popup, `storage` permission for `chrome.storage.sync`.
- The extension ID, name ("Shruggie Markdown"), and description must match the product identity (§2.1).

**Content Script**
- `extension/content.ts`: detect when the browser navigates to a URL ending in a recognized markdown extension. Intercept the raw content, inject the Shruggie Markdown preview UI, and render the content using the shared rendering pipeline.
- The injected UI must include the toolbar, full-view preview, and the option to toggle into split-view.

**Browser Action Popup**
- `extension/popup.html` and associated script: provide a URL input field for pasting a remote markdown URL. On submit, open a new tab with the rendered content.
- Include a link to the settings view.

**Platform Adapter**

## Notes for the Orchestrating Agent

1. **Sub-agent context windows.** Each sprint is scoped to fit a single agent context window. If a sprint is delegated to a sub-agent, provide it with the full `shruggie-md-spec.md` and this plan document. 2. **Parallelization opportunities.** Sprints 1 through 4 are strictly sequential. After Sprint 4 completes, Sprints 5 and 6 can run in parallel (the editor does not depend on the config system, and the config system does not depend on the editor). Sprint 7 depends on both 5 and 6. Sprint 8 depends on 6 and 7. Sprint 9's two sub-tasks (Chrome extension and PWA) can run in parallel with each other, and can begin as soon as Sprint 6 completes (they need the config adapters). Sprint 10 depends on everything.

4. **Testing discipline.** Each sprint's acceptance criteria define the minimum verification required before the sprint is considered complete. Sub-agents should run these checks and report results.

5. **File encoding.** All source files must be UTF-8 without BOM.
