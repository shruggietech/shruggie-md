### Task 1.2: NSIS Uninstaller Audit

Investigate the Tauri 2 NSIS installer/uninstaller behavior on Windows. Two issues have been reported:

**Issue A — Desktop shortcut not removed on uninstall.** The NSIS uninstaller is expected to remove all artifacts it created during installation, including any desktop shortcut. Investigate whether Tauri's NSIS configuration provides control over desktop shortcut creation and removal. Check `src-tauri/tauri.conf.json` under `bundle.windows.nsis` for relevant settings. If Tauri exposes a `desktopShortcut` or equivalent option, configure it. If the shortcut is created by the installer but the uninstaller does not remove it, this may be a Tauri bug; document the finding.

Relevant Tauri 2 NSIS config properties to check:
- `installerIcon`, `headerImage`, `sidebarImage`
- `installMode` (currently `"currentUser"`)
- Any shortcut-related properties

**Issue B — In-place upgrade keeps old code.** When installing a newer version over an existing installation, the registry version reportedly updates but the application binary may not be replaced. Investigate whether this is a Tauri NSIS `installMode` issue (e.g., install path mismatch between `currentUser` and `perMachine`), a WiX upgrade GUID conflict, or a stale-cache problem. Check:
- Whether the NSIS installer's `upgradeCode` or product GUID is stable across versions.
- Whether `installMode: "currentUser"` causes the new version to install to a different path than the old version.
- Whether the old process is still running and locking files during the upgrade attempt.

For both issues: if a fix is achievable through `tauri.conf.json` configuration changes, apply it. If the issue is in Tauri's NSIS template and cannot be fixed from configuration alone, document the root cause, any workarounds, and whether a Tauri GitHub issue should be filed.

## Phase 2 — Desktop Bug Fixes

### Task 2.1: PDF Export — Default Filename

**Problem:** When exporting to PDF, the browser print/save dialog opens with an empty filename field instead of a helpful default.

**Context:** PDF export is handled by `src/hooks/usePdfExport.ts`. The current implementation compiles markdown to HTML, injects it into a hidden iframe, and calls `window.print()` on the iframe. The print dialog's default filename is typically derived from the document `<title>` element.

**Fix:** In `usePdfExport.ts` (or the HTML generation path it uses), set the `<title>` of the generated HTML document to a meaningful default. Use the current filename (without extension) if a file is open, or `"Untitled"` if no file is open. This requires `usePdfExport` to accept the current filename as a parameter. Update the call site in `App.tsx` accordingly.

Verify that the HTML export path (`useHtmlExport.ts`) already handles this correctly (it appears to use `"Shruggie MD Export"` as the title); the PDF path should follow a similar pattern but use the actual document name.

### Task 2.2: PDF Export — Double Save Dialog

**Problem:** After saving a PDF via the print dialog, the dialog reappears and offers a second save session. Neither the HTML nor Markdown export paths exhibit this behavior.

**Context:** The PDF export calls `iframeWindow.print()` on a dynamically created hidden iframe. The double-dialog behavior suggests one of:
1. The `print()` call is being invoked twice (check for duplicate event handlers or re-renders).
2. The iframe's `afterprint` or `onafterprint` event is triggering a second print cycle.
3. The iframe is not being cleaned up before a second render cycle triggers another print.

**Fix:** Investigate `usePdfExport.ts` and trace the full lifecycle of the iframe from creation through `print()` to cleanup. Ensure:
1. The iframe is created once per export invocation.
2. `print()` is called exactly once.
3. An `afterprint` listener (or a timeout fallback) removes the iframe from the DOM after the dialog closes.
4. No React re-render or state update during the print flow triggers a second invocation.

Add a guard flag (e.g., `isPrinting`) to prevent re-entrant calls if necessary.

### Task 2.3: Workspaces — Empty State Message and New File Button

**Problem:** The default workspace, when empty, displays: "No files found. Add a workspace to get started." This message is misleading because adding a workspace is not the only way to populate the view; the user can also create a new file.

**Fix:** In `src/components/Workspaces/Workspaces.tsx`:

1. Change the empty state message to: "No files found. Create a new file or add a workspace to get started."
2. Add a "New File" button alongside the existing "New Workspace" button in the empty state area. This button should trigger the same `onNewDocument` handler used by the toolbar's New Document button, switching the user to edit mode with a blank buffer. The `onNewDocument` callback must be passed into the Workspaces component (add it to the props interface if not already present).

### Task 2.4: Workspaces — Files Not Listed After App Restart

**Problem:** After closing and reopening the application, navigating to the Workspaces view shows no files within workspaces, even after clicking "Refresh."

**Root cause investigation:** The `useWorkspaces` hook auto-scans when the active workspace changes (tracked via `prevWsId` ref). After an app restart, the active workspace ID is reset to `null` in React state even if a default workspace exists in the database. The initialization effect sets `activeWorkspaceId` but the scan may not trigger if the workspace ID assignment and the `activeWorkspace` derivation happen in the same render cycle without triggering the `useEffect` that watches `activeWorkspace`.

**Fix:** In `src/hooks/useWorkspaces.ts`:
1. After the initialization effect loads workspaces and sets `activeWorkspaceId`, explicitly trigger a scan once initialization completes (rather than relying solely on the `prevWsId` ref comparison in a separate effect).
2. Ensure the `refreshWorkspaces` function (called by the Refresh button) also triggers a file scan after reloading the workspace list, not just a workspace list reload.

Add a regression test: mock the storage adapter to return a workspace list with one workspace, verify that `scan` is invoked after initialization completes.

### Task 2.5: Workspaces — Restore Recursive Toggle UI

**Problem:** The per-workspace "Recursive traversal" toggle (spec §9.6) was removed from or is broken in the workspace settings UI. The setting still exists in the data model and the `useWorkspaces` hook still reads `wsSettings.recursive`, but the user has no way to change it.

**Fix:** Locate where the per-workspace settings modal was previously rendered (likely triggered by an ellipsis/gear button on each workspace in the workspace list panel). If the modal component still exists but is disconnected, reconnect it. If it was removed, recreate it as a simple modal with toggles for:
- Recursive traversal (boolean, default `true`)
- Show hidden files (boolean, default `false`)
- Independent extension rules (boolean, default `false`)
- Extension whitelist (string array, visible only when independent extension rules is enabled)

These are the four settings defined in spec §9.6. The modal should read from and write to the workspace's `settings` JSON column via the existing `updateWorkspace` storage method.

Verify the toggle is accessible from the Workspaces view and that changing the recursive setting causes a rescan of the affected workspace.

## Phase 3 — Chrome Extension Dark Mode

### Task 3.1: Investigate Dark Mode Theming

**Problem:** When switching to dark mode in the Chrome Extension, text/fonts transition to dark-mode colors but the background does not change. Reported by a tester; the affected surface (content script overlay, browser action popup, or both) is unknown.

**Investigation scope — content script (`extension/content.ts`):**
1. The content script injects a shadow DOM with inline CSS. Check whether the `[data-theme="dark"]` selector is being applied to the shadow host or the shadow root.
2. Verify that the `background-color` property on the shadow host or its inner wrapper references `var(--color-bg-primary)` and that this variable resolves correctly under the `[data-theme="dark"]` selector within the shadow DOM.
3. Shadow DOM encapsulates styles; `data-theme` attributes set on the host page's `<html>` or `<body>` element do not cascade into the shadow tree. The theme attribute must be set on an element inside the shadow DOM (or on the shadow host itself if the CSS selectors target `:host([data-theme="dark"])`).

**Investigation scope — browser action popup:**
1. If the popup uses the same React app entry point as the main application, the theming should work identically. Check whether the popup's HTML template includes the correct `data-theme` attribute or whether the `ThemeProvider` is mounted.
2. If the popup is a minimal standalone HTML page, verify it applies theme tokens correctly.

**Fix:** Apply the necessary CSS or attribute corrections to ensure both surfaces respond to dark mode. If the content script's shadow DOM is the issue, the fix likely involves setting `data-theme` on the shadow host element and adjusting selectors. If the popup is the issue, ensure the `ThemeProvider` initializes and applies the `data-theme` attribute on mount.

Add a note in the after-action report documenting which surface was affected and what the root cause was.

## Phase 4 — State Persistence

### Task 4.1: Persist and Restore Last-Viewed Document

**Problem:** When the application is closed and reopened, it launches with a blank editor instead of restoring the document the user was last viewing.

**Implementation:**

1. Add two new config keys to the general settings namespace:
   - `general.lastDocumentPath` (`string | null`, default `null`): The filesystem path of the last-opened local document.
   - `general.lastDocumentSource` (`"local" | "remote" | null`, default `null`): Whether the last document was local or remote. Remote documents are not restored (they require a fresh fetch).

2. In `App.tsx`, when a file is successfully opened (via Open dialog, workspace file click, CLI argument, or Save As), persist the file path to `general.lastDocumentPath` via the config update mechanism. When the source is remote, set `general.lastDocumentSource` to `"remote"`. When it is a local file, set it to `"local"`.

3. On app startup (in the initialization flow of `App.tsx`), after config and storage are loaded:
   - Read `general.lastDocumentPath` and `general.lastDocumentSource`.
   - If the source is `"local"` and the path is non-null, attempt to read the file via the platform adapter's `readFile`.
   - If the read succeeds, populate `content`, `fileName`, `filePath`, and start the file watcher, exactly as `handleWorkspaceFileSelect` does today.
   - If the read fails (file deleted, moved, or inaccessible), clear the persisted keys and proceed with the normal blank-editor startup.
   - If the source is `"remote"` or `null`, do not attempt restoration.

4. When `handleNewDocument` is called (user creates a blank document), clear `general.lastDocumentPath` and `general.lastDocumentSource`.

### Task 4.2: Persist and Restore Last Active View

**Context:** The `general.lastViewMode` config key already exists and persists the last content view mode (`"view"`, `"edit"`, `"edit-only"`). However, the current implementation only persists content modes; if the user was in the Workspaces or Settings view, those are not restored.

**Enhancement:** Extend the view restoration logic so that if the user's last active view before closing was Workspaces or Settings, the application reopens to that view. Update the `handleViewChange` function to persist all five view mode values (`"view"`, `"edit"`, `"edit-only"`, `"workspaces"`, `"settings"`), not just the three content modes.

On startup, if the restored view mode is `"workspaces"` or `"settings"`, navigate directly to that view. The last-document restoration from Task 4.1 should still load the document in the background so it is ready when the user switches to a content view.

Update the spec §10.1.1 description of `general.lastViewMode` to reflect that all five values are now persisted.

## Phase 5 — Welcome Document Redesign

### Task 5.1: Redesign the Default Welcome Markdown

**Problem:** The default "Welcome to Shruggie MD" document shown on first launch is plain and lacks branding or useful links.

**Requirements for the new welcome document content:**

1. Include the application icon/logo. Use a markdown image reference pointing to a relative path or a known asset location. If no web-accessible logo URL exists, use an inline base64-encoded small version of the icon, or reference the icon from the assets directory. (Check `assets/brand/` for available logo files.)
2. Include attribution to Shruggie LLC (DBA ShruggieTech) as the company building the application.
3. Include clickable hyperlinks to:
   - The GitHub repository: `https://github.com/shruggietech/shruggie-md`
   - The repository README: `https://github.com/shruggietech/shruggie-md#readme`
4. Include a brief feature overview (3-5 bullet points covering the key capabilities: multiple rendering engines, workspaces, theming, cross-platform support, export options).
5. Keep the tone friendly and concise. This is the first thing a user sees; it should feel welcoming without being verbose.

The welcome document is the initial value of the `content` state in `App.tsx` when no file is open. Locate where this default content string is defined and replace it.

### Task 5.2: Add Info Toolbar Button

**Problem:** The About section is currently buried inside the Settings view. It should be more accessible via a dedicated toolbar button.

**Implementation:**

1. Add an "Info" button to the toolbar, positioned next to the "Settings" button (gear icon). Use an appropriate icon (e.g., `Info` from `lucide-react`).

2. The Info button opens a dropdown menu (not a view switch) with two items:
   - **About** — Opens the About view (see Task 5.3).
   - **Help** — Opens the Help view (see Task 5.4).

3. The dropdown should use the same visual language as other dropdowns/menus in the application (consistent with the SplitButton dropdown pattern or a simple positioned menu).

### Task 5.3: Move About to Standalone View

**Problem:** The About section currently lives as §7 within the Settings panel. It should be its own view, accessible from the Info dropdown.

**Implementation:**

1. Create a new view mode value: extend `ViewMode` to include `"about"`. The About view is a transient view (like Settings and Workspaces), not a content mode; it should not be persisted to `general.lastViewMode`.
   - **Correction:** To avoid proliferating view modes, implement About as a modal dialog instead of a full view. The Info > About menu item opens a centered modal. This is more consistent with how "About" works in most desktop applications.

2. The About modal should display:
   - Application name ("Shruggie Markdown")
   - Application icon/logo
   - Current version (from `__APP_VERSION__` build-time constant)
   - Build target (Tauri/Chrome Extension/PWA, from platform detection)
   - Attribution: "Built by Shruggie LLC (DBA ShruggieTech)"
   - Link to the GitHub repository
   - License: "Apache-2.0"

3. Remove the About section (§7) from the Settings panel in `src/components/Settings/Settings.tsx`. Settings should end at the Advanced section.

4. Create `src/components/AboutModal/AboutModal.tsx` and `src/components/AboutModal/index.ts`.

### Task 5.4: Create Standalone Help View

**Problem:** There is no built-in help reference. The welcome document content should be accessible as a read-only help page at any time.

**Implementation:**

1. Add a `"help"` view mode to `ViewMode`. The Help view is a transient view; do not persist it to `general.lastViewMode`.

2. The Help view renders the same content as the redesigned welcome document (Task 5.1), but:
   - It is **not editable**. Render using the Preview component only (no editor).
   - It is **not treated as an open document**. It does not set `fileName`, `filePath`, or `content` state. It does not affect the file watcher. It does not appear as a "document" that can be saved or exported.
   - It uses its own static content string, imported from the same source as the welcome document default.

3. The toolbar should show a minimal state when the Help view is active (similar to Settings view behavior): no Save, Export, or file-related controls.

4. Wire the Info > Help menu item to switch to the Help view via `handleViewChange("help")`.

5. Extract the welcome document content string into a shared constant (e.g., `src/constants/welcomeContent.ts`) so both the initial `App.tsx` default and the Help view import from the same source. This ensures they stay in sync.
