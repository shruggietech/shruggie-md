## Summary

This sprint addresses a cluster of owner-reported defects and UX refinements covering Windows file association launch behavior, markdown rendering fidelity (bullets, inline code color, engine-switch observability, GitHub-default alignment), a new Refresh toolbar control with unsaved-changes guard, the Preview font-family setting (constrained dropdown with persistence), and toolbar spacing cleanup. Work is organized into seven phases executed sequentially.

**Scope boundaries:** This plan does not authorize new rendering engines beyond those already wired, new export formats, workspace schema changes, syntax highlighting changes inside fenced code blocks, or any modification to the CI/CD pipeline. Font selection is limited to a curated static list. If the Markdown Engine dropdown investigation reveals that the engines are functionally identical by design, document that finding rather than inventing behavioral differences.

## Phase 2 — Windows File Association Launch Fix

### Task 2.1: Diagnose the Launch Argument Path

Investigate how Shruggie Markdown receives a file path when launched via the Windows shell "Open With" verb. Inspect `src-tauri/src/main.rs` (or equivalent entry) and any existing CLI/argument handling. Tauri 2 typically exposes launch arguments via `tauri_plugin_cli` or `std::env::args()` in the setup hook, and requires the file path to be forwarded to the frontend before the initial document-load routine decides whether to show the welcome document.

Determine whether the current code:

- Reads `argv[1]` (or equivalent) at startup.
- Forwards that path through to the frontend via an event, IPC command, or `window.__INITIAL_FILE__` global.
- Is being overridden by the welcome-document restoration logic added in v0.3.6.

### Task 2.2: Implement the Fix

Wire the startup argument path so that:

1. If the app is launched with a file path argument and the file exists, that file is loaded as the initial document.
2. The welcome document and last-session restoration logic are both bypassed when a launch-path argument is present.
3. Non-existent or unreadable paths fall back to the normal startup flow (last-session restoration, then welcome document).

Verify that the `tauri.conf.json` `bundle.windows.fileAssociations` block (if present) is intact. Do not remove or restructure it.

### Task 2.3: Test Coverage

Add a unit or integration test for the frontend launch-path handler that covers: path present and valid, path present and missing, and path absent. Rust-side argument parsing may be verified by manual smoke test during the human gate.

## Phase 3 — Markdown Rendering Fidelity

### Task 3.1: Bullet Glyph Restoration

Investigate why `ul > li` elements are rendering without bullet markers in View and Edit modes. The most likely causes are a global CSS reset removing `list-style`, a Tailwind/preflight rule, or a preview stylesheet override in `src/styles/preview.css`. Fix the root cause rather than patching symptoms. Ensure both unordered and ordered lists render with standard markers at all nesting levels.

### Task 3.2: Inline Code Color

Apply a distinct color to inline code spans (single-backtick `<code>` elements that are not inside a `<pre>` block). Use the muted Shruggie green variant already defined in the design tokens if available; otherwise define a new token `--color-inline-code` and document it. This rule must not affect fenced code blocks, which retain their existing syntax highlighting treatment. Apply the color in both light and dark themes with appropriate contrast.

### Task 3.3: GitHub-Aligned Default Preview Style

Audit `src/styles/preview.css` against GitHub's rendered markdown defaults. Key alignment points:

- Base font stack and size
- Heading sizes, weights, and bottom borders on `h1`/`h2`
- Blockquote left border and muted text color
- Table borders, header row background, and zebra striping
- Link color
- Horizontal rule styling
- Paragraph spacing

Adjust `preview.css` to match conventional GitHub appearance. Do not introduce a hard dependency on GitHub's CSS; reproduce the visual conventions using the existing design tokens where possible.

### Task 3.4: Markdown Engine Dropdown Audit

Inspect the Markdown Engine setting end-to-end. Trace the dropdown value from Settings through to the rendering pipeline and confirm whether switching engines actually routes to a different parser/renderer. Three outcomes are possible:

1. The engines are functionally wired but produce near-identical output on common markdown. Document this in the spec and add a small visible difference-surface (for example, a feature matrix note in the Settings description text).
2. The dropdown value is read but not used. Fix the routing so the selected engine is actually invoked.
3. Only one engine is installed. Document the current state honestly and do not fabricate engine options.

Record your findings in the sprint notes and reflect the outcome in §5.x of the spec.

## Phase 4 — Refresh Button and Unsaved-Changes Guard

### Task 4.1: Add Refresh Toolbar Button

Add a new toolbar button immediately to the right of the Save control. Use a circular-arrow refresh glyph consistent with the existing toolbar icon set (lucide or equivalent). The button's tooltip should read "Refresh preview".

Clicking Refresh with no unsaved changes forces a re-render of the current document from its on-disk (or in-memory) source and clears any stale preview state.

### Task 4.2: Unsaved Changes Modal

If the document has unsaved edits when Refresh is pressed, display a modal with the title "Unsaved Changes" and body text explaining that refreshing will discard unsaved edits. Provide two actions:

- **Refresh** (destructive): discards unsaved edits and re-renders.
- **Cancel** (default): closes the modal and takes no action.

Use the existing modal component pattern from the About modal (v0.3.6) for visual consistency.

### Task 4.3: Test Coverage

Add tests covering: Refresh with clean document triggers re-render, Refresh with dirty document opens modal, modal Cancel is a no-op, modal Refresh discards edits and re-renders.

## Phase 5 — Font Family Setting Dropdown

### Task 5.1: Curate the Font List

Define a static list of font families in a new module `src/config/previewFonts.ts`. The list should include widely available system and web-safe fonts with human-friendly display names, sorted alphabetically. Suggested baseline set (finalize during implementation):

- Arial
- Cambria
- Consolas
- Courier New
- Georgia
- Helvetica
- Inter
- JetBrains Mono
- Roboto
- System Default (maps to the existing default stack)
- Times New Roman
- Verdana

Each entry is a `{ label: string, value: string }` pair where `value` is the CSS `font-family` declaration.

### Task 5.2: Replace the Free-Text Input

Replace the current free-text input in Settings > Preview > Font family with a `<select>` populated from the curated list. The default selection on first run is "System Default". Remove any code paths that allowed arbitrary user-entered strings.

### Task 5.3: Persistence

Confirm that the selected font family persists across sessions via the existing settings store. If the current storage key is already persistent, verify the round-trip. If not, add the key to the persisted settings shape and ensure migration for existing installs falls back to "System Default" silently.

### Task 5.4: Test Coverage

Add or update tests covering: default selection on first run, selection persistence round-trip, and application of the selected font to the preview surface.
