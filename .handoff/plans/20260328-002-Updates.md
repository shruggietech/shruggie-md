## Phase 2 — Playwright MCP Configuration

**Goal:** Set up the Playwright MCP server for VS Code Copilot so that agents operating in VS Code can launch the running application and take screenshots for visual verification.

### Task 2.1: Create MCP Configuration

Create or update `.vscode/mcp.json` with the Playwright MCP server entry. If the file already exists (from the workspace settings), merge the new entry without overwriting existing content.

```json
{
  "servers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}
```

**Note:** If `.vscode/mcp.json` already exists with other content, add the `playwright` key under `servers` alongside existing entries. Do not replace the file wholesale.

### Task 2.2: Install Playwright as a Dev Dependency

```bash
pnpm add -D @playwright/test
npx playwright install chromium
```

Only Chromium is needed. The MCP server uses it to drive browser-based screenshot verification. Playwright's other browsers (Firefox, WebKit) are not required for this use case.

### Task 2.3: Update CLAUDE.md

Add a new subsection under §5 (Known Agent Failure Modes) or as a new §6:

```markdown
## 6. Visual Verification via Playwright MCP

When operating in VS Code with the Playwright MCP server connected, agents
MUST use it to visually verify UI changes. The workflow is:

1. Start the dev server: `pnpm tauri:dev` (or `pnpm run dev` for web-only).
2. Use the Playwright MCP `browser_navigate` tool to open `http://localhost:1420`.
3. Use `browser_snapshot` or `browser_take_screenshot` to capture the current UI.
4. Confirm the rendered output matches the expected changes before reporting
   a task as complete.

Screenshot verification is mandatory for any task that modifies:

- Component layout or structure
- CSS / styling (including theme tokens)
- Toolbar, dialog, or panel content
- View mode behavior

Typecheck and test passage alone are not sufficient evidence that UI changes
are correct. If the Playwright MCP server is not available, note this in the
after-action report and flag the UI verification as pending.
```

Preserve the existing section numbering. If §6 already exists for a different purpose, use the next available number.

### Task 2.4: Update .github/copilot-instructions.md

Add a brief note referencing the Playwright MCP server and the visual verification requirement:

```markdown
## Visual Verification

This project has the Playwright MCP server configured in `.vscode/mcp.json`.

Review all changes. Commit with:

```

### H2: Windows Smoke Test

Perform a manual smoke test on Windows:

1. `pnpm tauri:build` — confirm NSIS and MSI installers are produced.
2. Install via the NSIS installer.
3. Launch the installed application (not dev mode).
4. Verify: application opens, toolbar renders, view mode switching works.
5. Verify: Open dialog (File tab and URL tab) functions correctly.
6. Verify: Edit mode, save, export (HTML, PDF).
7. Verify: Workspace creation and file listing.
8. Verify: Settings panel opens, changes persist.

If any step fails, do not proceed. File a defect and create a new sprint plan.

### H3: Finalize Changelog Date

If the agent left the date as `YYYY-MM-DD`, replace it with the actual release date.

### H4: Tag and Push

```bash
git tag v0.3.1
git push origin main
git push origin v0.3.1
```

Pushing the tag triggers the GitHub Actions release workflow (`release.yml`), which builds all three desktop platforms, packages the Chrome extension, and publishes a GitHub Release.

### H5: Verify Release Artifacts

After the release workflow completes:

1. Confirm the GitHub Release page lists all expected artifacts (Windows MSI, Windows NSIS installer, macOS DMG, Linux AppImage, Chrome extension zip).
2. Download the Windows installer and confirm it matches the local smoke test.
3. If any platform build fails, investigate via the Actions run log. macOS and Linux builds have not been manually tested but should compile if the CI environment is correct.
