## Phase 2 — Critical Bug Fixes

**Goal:** Fix functional bugs that break existing features without changing layout or adding new features.

### Task 2.1: Fix Workspace Path Construction in OpenDialog

**Root cause:** In `src/components/OpenDialog/OpenDialog.tsx`, the `handleCreateWorkspace` function constructs the workspace path as:

```ts
const wsPath = `${appDataDir}workspaces/${trimmedName}/`;
```

This is missing a path separator between `appDataDir` and `workspaces`. The Tauri `appConfigDir()` function does not guarantee a trailing separator on all platforms.

**Fix:** Use the same path construction pattern as `useWorkspaces.ts`:

```ts
const wsPath = `${appDataDir}/workspaces/${trimmedName}`;
```

Also normalize any double separators that could result if `appDataDir` does return a trailing slash. A safe approach:

```ts
const base = appDataDir.endsWith("/") || appDataDir.endsWith("\\")
  ? appDataDir
  : `${appDataDir}/`;
const wsPath = `${base}workspaces/${trimmedName}`;
```

Apply the same normalization in `useWorkspaces.ts` → `createWorkspace` for consistency, so both code paths produce identical results for the same workspace name.

### Task 2.2: Fix SplitButton Label Visibility

**Root cause:** The `SplitButton` component in `src/components/common/SplitButton.tsx` does not accept a `showLabel` or `showButtonLabels` prop, so it never renders a text label for Save/Save As regardless of the user's label visibility setting.

**Fix:**

1. Add a `showLabel?: boolean` prop to `SplitButton`.
2. When `showLabel` is true and a file path is established, render a "Save" text label inside the primary button region (matching the pattern used by `Button`).
3. When `showLabel` is true and no file path is established (Save As mode), render a "Save As" text label.
4. Update the `Toolbar` component to pass `showButtonLabels` through to `SplitButton`.
5. Add a test case to `design-system.test.tsx` confirming the label renders when `showLabel` is true.

### Task 2.3: Investigate and Fix Tooltip Rendering

**Current state:** The `Tooltip` component (`src/components/common/Tooltip.tsx`) exists, has a 400ms delay, and is wired into every `Button` via the `tooltip` prop. Tests in `design-system.test.tsx` confirm the tooltip appears after the delay in the JSDOM environment.

**Investigation:** The tooltip renders using `position: fixed` and `z-index: 9999`. Possible failure modes in the Tauri WebView:

1. The tooltip renders but is clipped by an `overflow: hidden` ancestor.
2. The `position: fixed` coordinates are incorrect in the Tauri webview context.
3. The tooltip wrapper `div` with `display: inline-flex` is breaking the event propagation for `mouseEnter`/`mouseLeave`.

**Steps:**

1. Use Playwright MCP to hover over a toolbar button and wait 500ms. Take a screenshot. Check if a tooltip is visible.
2. If not visible, use Playwright's `browser_console` (if available) or add a temporary `console.log` inside `showTooltip` to confirm the timer fires.
3. Check for `overflow: hidden` or `overflow: clip` on any ancestor of the toolbar buttons that could clip fixed-position tooltips. The toolbar `<header>` has `height: 40px` and `maxHeight: 40px`, which combined with any overflow restriction would clip tooltips that position above the toolbar.
4. If the toolbar's `overflow` is the culprit, the fix is to remove `overflow: hidden` from the toolbar header (it should not need overflow clipping, since its children are flex-laid-out with known sizes).
5. If the issue is coordinate accuracy, switch the tooltip from `position: fixed` to a portal-based approach using `ReactDOM.createPortal` rendering into `document.body`. This sidesteps all ancestor clipping issues.
6. After applying the fix, verify with Playwright MCP that tooltips now appear on hover.

### Task 2.4: Fix Settings Panel Scrollbar Position

**Root cause:** The `Settings` component renders a single `<div>` with `maxWidth: 680px`, `margin: "0 auto"`, and `overflow: auto`. The scrollbar attaches to this centered div, so it appears at the 680px boundary (visually mid-window) rather than at the right edge of the application window.

**Fix:** Restructure the Settings layout as two nested containers:

1. **Outer container** (full width): `flex: 1`, `overflow-y: auto`, `width: 100%`. This is the scrollable element, so the scrollbar hugs the right window edge.
2. **Inner container** (centered content): `maxWidth: 680px`, `margin: 0 auto`, `padding: var(--space-6)`. This contains all the settings sections but does not scroll.

The outer container can be added as a wrapper `<div>` inside the `Settings` component return, or handled in `App.tsx` at the layout level. The important thing is that the element with `overflow: auto` spans the full viewport width.

## Phase 3 — Toolbar Redesign

**Goal:** Restructure the toolbar into clearly defined visual zones that separate navigation, document context, and document actions. Add a status bar at the bottom of the application window.

### Design Rationale

The current toolbar places all elements in a flat sequence: app name, five mode buttons, chevron, filename, action buttons, status text. This creates three problems:

1. **No visual hierarchy.** Navigation buttons (View, Edit, Edit Only, Workspaces, Settings) are visually identical to action buttons (Open, New, Save, Export). Users cannot scan the toolbar and quickly distinguish "where am I" from "what can I do."
2. **Mismatched grouping.** Workspaces and Settings are navigation destinations that replace the entire content area. View, Edit, and Edit Only are content modes that change how you see your document. Grouping all five as equivalent items in a single segmented control obscures this distinction.
3. **No breathing room for the filename.** The filename is squeezed between mode buttons and action buttons with no visual separation. Long filenames overflow without any truncation strategy.

The redesign follows the established **Navigation | Context | Actions** pattern used by VS Code, Obsidian, and most modern document editors.

### Task 3.1: Implement Three-Zone Toolbar Layout

Restructure the `Toolbar` component into three visually distinct zones separated by gaps (using `var(--space-4)` or wider).

**Left Zone — Navigation:**

- **Content mode segmented control:** View | Edit | Edit Only. These three remain as a tightly grouped set with `--color-bg-active` highlighting the active mode. The segmented control gets a subtle `1px solid var(--color-border-subtle)` border around the group to visually contain it.
- **Quick settings chevron:** Immediately after the segmented control. This stays here because the pop-down panel content is mode-dependent.
- After the chevron, a visual gap (not a drawn line) separates the left zone from the center zone.

**Center Zone — Document Context:**

- **File name display:** Centered within the remaining space using `flex: 1` with `text-align: center`. Uses `text-overflow: ellipsis`, `overflow: hidden`, `white-space: nowrap`. Maximum width is constrained by the space between the left and right zones.
- A `title` attribute on the filename `<span>` shows the full file path on native hover (immediate, not dependent on the custom Tooltip component).
- When no file is loaded, displays "No file" in `--color-text-tertiary` (more muted than the current `--color-text-secondary`).
- The "Shruggie MD" app name is removed from the toolbar. It adds no utility (the window title bar already identifies the application). Attribution moves to the About section (Phase 5).

**Right Zone — Actions and Destinations:**

- **File action buttons:** Open, New, Save/SaveAs (SplitButton), Export. Grouped together.
- **Visual separator:** A `1px` vertical line using `--color-border-subtle`, height ~20px, with `var(--space-2)` horizontal margin.
- **Destination buttons:** Workspaces (FolderOpen icon, shown only when `hasFilesystem` is true) and Settings (Settings gear icon). These are standard `Button` components, not part of the segmented control. When one of these destinations is active, it receives the same `--color-bg-active` highlight.

**When Workspaces view is active:** The center zone switches to workspace-specific controls (Refresh button, filter input, scanning status), as it does now. File action buttons in the right zone are hidden (since no document is open in this view). The Workspaces destination button remains highlighted.

**When Settings view is active:** The center zone is empty or shows "Settings" as a heading. File action buttons in the right zone are hidden. The Settings destination button remains highlighted.

### Task 3.2: Update ViewMode Segmented Control Styling

The segmented control (View | Edit | Edit Only) needs a containing visual boundary to read as a single grouped control:

- Container: `display: inline-flex`, `border-radius: var(--radius-sm)`, `border: 1px solid var(--color-border-subtle)`, `background: var(--color-bg-tertiary)`, `padding: 2px`.
- Active button: `background: var(--color-bg-active)`, `border-radius: calc(var(--radius-sm) - 2px)`.
- Inactive buttons: transparent background.

This follows the standard segmented control pattern (similar to iOS/macOS segmented controls) and clearly communicates that these buttons form a mutually exclusive group.

### Task 3.3: Add Status Bar Component

Create a new `StatusBar` component (`src/components/StatusBar/StatusBar.tsx`) rendered at the bottom of the application window.

**Layout:**

- Height: `24px`. Background: `--color-bg-secondary`. Top border: `1px solid var(--color-border-primary)`.
- Font: `--font-size-xs`, `--font-ui`, `--color-text-tertiary`.
- Two regions: left-aligned and right-aligned, using flexbox with `justify-content: space-between`.

**Content:**

- **Left:** Active markdown engine name (read from `config.engine.activeEngine`). Displayed as the engine's display name (e.g., "markdown-it" → "markdown-it (GFM)", "marked" → "Marked", "remark" → "Remark").
- **Right:** "Shruggie Markdown v{version}" where `{version}` is read from a build-time constant (see Task 3.4).

The status bar is always visible across all view modes.

### Task 3.4: Expose Version as a Build-Time Constant

The application version needs to be available at runtime for the status bar and the About section (Phase 5).

**Approach:** Use Vite's `define` option in `vite.config.ts` to inject the version from `package.json`:

```ts
import { readFileSync } from "fs";

const pkg = JSON.parse(readFileSync("./package.json", "utf-8"));

export default defineConfig({
  define: {
    __APP_VERSION__: JSON.stringify(pkg.version),
  },
  // ... existing config
});
```

Add a type declaration in `src/vite-env.d.ts` (or a new `src/globals.d.ts`):

```ts
declare const __APP_VERSION__: string;
```

This constant is then usable anywhere in the frontend code as `__APP_VERSION__`.

```
└─────────────────────────────────────┘
```

The content area should be `flex: 1` with `overflow: hidden` (or `auto`, depending on the view). The status bar is outside the scrollable content.

## Phase 4 — Workspace UX Improvements

**Goal:** Fix workspace file listing so the Workspaces view actually displays files, add workspace creation directly in the Workspaces view (not just in the Open dialog), and investigate recursive discovery.

### Task 4.1: Debug Workspace File Listing Pipeline

The Workspaces view reports "no files found" even when the workspace directory contains markdown files. Investigate the full pipeline:

1. **Confirm the workspace path is correct.** Add a temporary `console.log` (or use Playwright console tools) to print `activeWorkspace.path` when the scan runs. Confirm this path exists on disk and contains `.md` files.
2. **Confirm `listDirectory` returns entries.** Log the return value of `adapter.listDirectory(wsPath, wsSettings.recursive)` to see what Tauri's `readDir` + `stat` pipeline produces.
3. **Check extension filtering.** The scan filters files by `activeExtensions` (from either per-workspace settings or global `fileExtensions.recognized`). Log the active extensions list and confirm `.md` is included.
4. **Check the `isDirectory` detection.** The Tauri adapter uses `entry.isDirectory ?? false`. If `readDir` does not populate `isDirectory` for some entries (it depends on the Tauri version and platform), files could be misclassified as directories and skipped.

**Likely fixes:**

- If the workspace path is wrong (e.g., double separators, missing directory), apply the same path normalization from Task 2.1.
- If `isDirectory` is unreliable, fall back to checking `entryStats.isDirectory` or `entryStats.isFile` from the `stat()` call, which is more reliable.
- If entries are returned but all filtered out, check the extension list against actual file extensions on disk.

After identifying and fixing the root cause, verify with Playwright MCP that the Workspaces view displays files when a workspace directory contains `.md` files.

### Task 4.2: Add Workspace Creation to the Workspaces View

Currently, the only place to create a workspace is inside the Open dialog's workspace dropdown ("Add a new workspace..."). This is unintuitive. Users navigating to the Workspaces view expect to manage workspaces there.

**Implementation:**

1. Add an "Add Workspace" button to the workspace list panel in the `Workspaces` component. Use the `Plus` icon from Lucide with a "New Workspace" label.
2. Clicking the button opens a modal dialog (reusing the existing `Modal` component) with:
   - A text input for the workspace name.
   - A type selector: "Internal" (app-managed directory) or "External" (choose existing directory). Default to Internal.
   - For External type: a "Browse..." button that invokes `platform.openDirectoryDialog()`.
   - A "Create" accent button and a "Cancel" ghost button.
   - Name validation using the existing `validateWorkspaceName` function, with error messages displayed inline.
3. On successful creation, refresh the workspace list and select the new workspace.
4. Ensure the workspace creation logic uses the same normalized path construction as Task 2.1 (no path separator inconsistencies).

The Open dialog's inline workspace creation can remain as a convenience, but the Workspaces view is now the primary workspace management surface.

### Task 4.3: Investigate and Fix Recursive File Discovery

The spec (§9.7) states that workspaces recursively traverse subdirectories by default. The Tauri adapter's `listDirectory` implementation does support recursion (it calls `processDir` recursively when `entry.isDirectory && recursive`). However, it has been reported as broken.

**Investigation steps:**

1. Create a test workspace with a nested directory structure containing `.md` files at multiple levels.
2. Enable recursive traversal in the workspace settings (this is the default).
3. Trigger a scan and log the entries returned by `listDirectory`.
4. If recursion works in `listDirectory` but files don't appear in the UI, the issue is in the `useWorkspaces` hook's filtering logic.
5. If `listDirectory` itself doesn't recurse, the issue is in the `isDirectory` detection (see Task 4.1).

**Possible fix:** If `entry.isDirectory` from Tauri's `readDir` is unreliable, use the `stat()` result's `isDirectory` property instead:

```ts
const entryStats = await stat(entryPath);
const isDir = entryStats.isDirectory; // from stat, not from readDir
```

### Task 5.2: Audit Preview CSS for GitHub-Flavored Styling

The default markdown engine (`markdown-it`) already parses GFM syntax (tables, task lists, strikethrough, autolinks) via plugins. The rendering pipeline is GFM-compatible. The preview CSS, however, may not visually match GitHub's rendering conventions.

**Audit the following against GitHub's rendered markdown appearance:**

1. **Heading sizes and borders.** GitHub uses a bottom border on `h1` and `h2`. The spec already calls for this (`1px solid --color-border-subtle`). Confirm the implementation matches.
2. **Code blocks.** GitHub uses a `border-radius: 6px` background with padding. Compare against the current `--radius-sm` (4px) and `--space-4` (16px) padding.
3. **Task list checkboxes.** Confirm `[ ]` and `[x]` render as styled checkboxes, not raw text.
4. **Tables.** GitHub renders tables with alternating row backgrounds and bordered cells. Confirm the preview CSS includes table styling. If not, add styles for `table`, `th`, `td` with:
   - Border: `1px solid var(--color-border-primary)`.
   - Header row: `--color-bg-secondary` background, `--font-weight-semibold`.
   - Alternating body rows: transparent and `--color-bg-tertiary`.
   - Padding: `var(--space-2) var(--space-3)`.
5. **Blockquotes.** GitHub uses a left border with muted text. Compare against current styling.
6. **Horizontal rules.** Confirm `---` renders as a visible line.
7. **Images.** Confirm images render with `max-width: 100%` to prevent overflow.

Make adjustments to the preview stylesheet (likely in `src/styles/preview.css` or inline within the `Preview` component) to close any gaps. The goal is not pixel-perfect GitHub replication but visual familiarity for users accustomed to GitHub's rendered markdown.

### H1: Review and Commit

Review all changes. Commit with:

```
feat: toolbar redesign, workspace UX improvements, bug fixes — targeting v0.3.2
```

### H2: Windows Smoke Test

Perform a manual smoke test on Windows:

1. `pnpm tauri:build` — confirm NSIS and MSI installers are produced.
2. Install via the NSIS installer.
3. Launch the installed application (not dev mode).
4. Verify: toolbar displays three zones correctly (segmented control, centered filename, action buttons + destinations).
5. Verify: tooltips appear on all toolbar buttons after hover delay.
6. Verify: status bar visible at bottom with engine name and version.
7. Verify: Settings > About section displays version and attribution.
8. Verify: Settings panel scrollbar is at the right window edge.
9. Verify: Workspaces view shows "Add Workspace" button, file listing works.
10. Verify: creating a workspace from the Workspaces view succeeds.
11. Verify: creating a workspace from the Open dialog succeeds.
12. Verify: recursive file discovery works when enabled.
13. Verify: "Show button labels" affects all buttons including Save.
14. Verify: preview pane renders GFM content (tables, task lists) with GitHub-aligned styling.

If any step fails, do not proceed. File a defect and create a new sprint plan.

### H3: Finalize Changelog Date

Replace `YYYY-MM-DD` with the actual release date in `CHANGELOG.md`.

### H4: Tag and Push

```bash
git tag v0.3.2
git push origin main
git push origin v0.3.2
```

### H5: Verify Release Artifacts

After the release workflow completes, confirm the GitHub Release page lists all expected artifacts.
