## Summary

This plan covers the full scope of work required to reach v0.3.0. It is organized into five sequential phases with explicit dependency relationships. Phases may not be reordered. Within each phase, tasks are numbered and should be completed in the order listed unless explicitly marked as parallelizable.

The work introduces: a storage architecture migration (JSON config to SQLite on desktop, IndexedDB on Chrome/PWA), a workspace system replacing the current library/mount model, a restructured view mode system with three modes, a collapsible pop-down toolbar, a unified Open dialog replacing all file entry points, and an export system replacing the current conversion actions. It also includes a batch of bug fixes and UI polish items that carry no architectural risk and should be completed first.

## Baseline Assumptions

- v0.1.1 Windows hotfix has been smoke-tested and confirmed working.
- v0.2.0 is the current codebase (Save/Save As, SplitButton, platform-aware defaults, tooltips on Settings elements).
- 258 tests pass, zero TypeScript errors.
- The spec will require significant updates after each phase. Spec updates are included as explicit tasks within each phase rather than deferred to the end.

## Standing Rules (Apply to Every Phase)

These rules carry forward from prior sprint plans and are non-negotiable.

1. **No release tagging.** The agent must not tag a release or create a GitHub release. That decision is reserved for William after manual smoke testing.
2. **Verify before declaring done.** Every phase ends with a verification task. The agent must run the application (`pnpm tauri dev` at minimum) and confirm the changes work before reporting completion. Build-only verification is insufficient.
3. **Spec fidelity.** When a task references a spec section, the agent must read that section before implementing. Do not rely on memory or assumptions about spec content.
4. **No scope creep.** Complete only the tasks listed in the current phase. Do not "improve" adjacent systems or refactor code outside the phase scope.
5. **Existing tests must pass.** At the end of every phase, `pnpm run test` must report zero failures. If a change breaks existing tests, the agent must fix the tests as part of that phase (not defer them).
6. **CHANGELOG accuracy.** The agent must update `docs/changelog.md` (or `CHANGELOG.md` at repo root, whichever exists) with an `[Unreleased]` section reflecting actual changes made. Do not fabricate test counts or feature completions.

## Phase 1: Bug Fixes and UI Polish

**Dependencies:** None. Start here.
**Parallelizable:** Yes. Tasks 1.1 through 1.6 are independent and may be completed in any order or in parallel. Task 1.7 (verification) must run last.
**Estimated scope:** Small. Low architectural risk.

### Task 1.1 — Scrollbar Overhaul

The application's scrollbars have three problems:

1. They render in light-mode styling even when the app is in dark mode (glaring white against dark surfaces).
2. They are approximately twice as thick as they should be.
3. In the Settings view, the scrollbar abuts directly against the settings cards instead of aligning to the far right edge of the window.

Fix all three issues. Use CSS custom scrollbar styling (`scrollbar-width`, `scrollbar-color`, and/or `::-webkit-scrollbar` pseudo-elements) that respects the current color mode. The scrollbar should use the existing design token system: track color derived from `--color-bg-secondary`, thumb color derived from `--color-bg-active` or similar. Thin profile (`scrollbar-width: thin` or equivalent pixel width of roughly 6-8px). The Settings view scrollbar should be on the outer container, not within individual cards.

### Task 1.2 — Editor Settings Wiring

The following toggles in the Editor section of Settings have no visible effect when toggled:

- **Show line numbers:** Must toggle `lineNumbers` in the CodeMirror 6 configuration. When off, the gutter disappears. When on, line numbers display.
- **Word wrap:** Must toggle `EditorView.lineWrapping` in the CodeMirror 6 extensions. When word wrap is turned off, the editor must NOT display a horizontal scrollbar. Instead, content should simply extend beyond the visible area (the user scrolls the editor pane itself, not a nested scrollbar). Achieving this may require setting `overflow-x: hidden` on the CodeMirror scroll container while allowing the editor's internal scroll to handle horizontal movement cleanly, or simply using CodeMirror's native scroll behavior with the scrollbar visually suppressed. Test both states (on and off) and confirm the gutter and wrapping behavior matches expectations.

Also verify that the **Font Family** setting in the Editor section is wired correctly. Change it to a different font and confirm the editor pane's font actually updates.

### Task 1.3 — Library Refresh Button Focus State

In the library view, clicking the refresh button causes a highlight/focus border to appear around the button. This border persists after the mouse leaves the button (un-hover). The focus ring should only be visible during keyboard focus (`:focus-visible`), not after a mouse click. Fix the button's focus/active styles so that:

- Mouse click triggers the action but does not leave a persistent outline.
- Keyboard focus (Tab navigation) still shows the focus ring for accessibility.

This is likely a `:focus` vs `:focus-visible` issue. Apply this fix globally to all toolbar buttons while you are here, not just the refresh button.

### Task 1.4 — Tooltips on All Interactive Elements

The v0.2.0 changelog claims tooltips were added to Settings and UrlInput elements. However, the brain dump indicates many toolbar buttons still lack tooltips. The spec (§4.6) states: "Every interactive element (buttons, toggles, icon-buttons, extension-chip remove buttons) must be wrapped in a tooltip. No interactive control may exist without a discoverable label."

Audit every interactive element across the entire application (toolbar buttons, view mode toggles, save/export actions, library controls, URL input controls, settings controls). Any element missing a tooltip must receive one. Tooltip behavior per spec: 400ms hover delay, 120ms fade-in, `--color-bg-active` background, `--color-text-primary` text, `--font-size-xs`, `--radius-sm` border radius, `--space-1` vertical padding, `--space-2` horizontal padding.

### Task 1.5 — Button Labels Setting

Add a new setting in the Appearance section of Settings:

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| Show button labels | boolean | `true` | When enabled, small text labels are displayed alongside icons on toolbar buttons and other icon-based controls throughout the application. When disabled, only icons are shown (tooltips remain available on hover). |

This setting applies to toolbar buttons, view mode toggles, and action buttons across the application. It does not apply to Settings page controls (those always show their labels). The implementation should add a small text label below or beside each icon button that reads the button's tooltip text (or a shorter variant where the tooltip text is too long). When the setting is off, these labels are hidden via CSS (not removed from the DOM, so screen readers still see them).

Persist this setting in configuration under `appearance.showButtonLabels`.

### Task 1.6 — Spec Update (Phase 1)

Update `shruggie-md-spec.md` to reflect the changes made in this phase:

- §4.6: Add scrollbar styling requirements (color mode awareness, thin profile, token-based colors).
- §4.6: Add the button labels setting and its interaction with tooltips.
- §10.1.1 or §10.2: Add `appearance.showButtonLabels` to the settings table.
- §7.1: Clarify that `lineNumbers` and `lineWrapping` are reactive to settings changes (not just initial config).

## Phase 2: View Mode Restructure and Pop-Down Toolbar

**Dependencies:** Phase 1 complete.
**Parallelizable:** Tasks 2.1 and 2.2 are tightly coupled and should be done sequentially. Task 2.3 (pop-down toolbar) can begin after 2.1 is done.
**Estimated scope:** Medium. Significant UI restructuring but no backend changes.

### Task 2.1 — Three-Mode View System

Replace the current two-mode system (full-view, split-view) with three modes:

1. **View** (rendered preview only, the current full-view). This is the default on web/PWA/Chrome Extension.
2. **Edit** (split pane: editor left, preview right, with draggable divider). This is the default on desktop. The draggable divider already exists per the spec (§5.3). Verify it works. If it does not, fix it.
3. **Edit Only** (full-window CodeMirror editor, no preview pane). This is new.

The toolbar must display a clear mode switcher using word + icon buttons (not icon-only). The three options should be visually grouped and clearly indicate the current mode. Use a segmented control pattern (three adjacent buttons, the active one highlighted with `--color-accent` or `--color-bg-active`).

Suggested icon + label pairings (use Lucide icons from the existing dependency):
- View: `Eye` icon + "View"
- Edit: `Columns2` icon (or `PanelLeftClose`) + "Edit"
- Edit Only: `FileEdit` icon (or `SquarePen`) + "Edit Only"

When `appearance.showButtonLabels` is off, these labels hide but the icons remain. The segmented control should still be visually coherent in icon-only mode.

Update the `general.lastViewMode` config key to support three values: `"view"`, `"edit"`, `"edit-only"`. The previous values `"full-view"` and `"split-view"` should be treated as aliases during config migration: `"full-view"` maps to `"view"`, `"split-view"` maps to `"edit"`.

Library and Settings remain transient navigation overlays. They do not appear in the mode switcher. Navigating to Library or Settings temporarily replaces the content area. Returning from Library or Settings restores the previously active mode.

### Task 2.2 — Edit Only Mode Implementation

The "Edit Only" mode renders the CodeMirror editor at full viewport width (minus toolbar height). No preview pane, no divider. The editor should respect all existing editor settings (font family, font size, line height, line numbers, word wrap, linting). The editor fills the same content area that the preview pane uses in View mode.

Save and Save As keyboard shortcuts must work in Edit Only mode identically to how they work in Edit (split) mode.

### Task 2.3 — Pop-Down Toolbar

Add a collapsible settings panel that drops down from the main toolbar when the user clicks a chevron/caret button (e.g., `ChevronDown` icon) in the toolbar. This panel is a horizontal or grid layout exposing quick-access settings. It slides down with a smooth animation (150ms, ease-out) and can be re-collapsed by clicking the same button (which rotates to `ChevronUp` when the panel is open).

The panel's visibility state is persisted in application config (`general.editorToolbarExpanded`, default `false`) and restored on launch.

The content of this panel varies by active view mode:

**View mode:**
- Preview font size (number input or slider)
- Preview line height (number input or slider)

**Edit mode (split pane):**
- Editor section (visually labeled "Editor"): Font size, line height, show line numbers toggle, word wrap toggle, linting toggle, linter selection dropdown
- Preview section (visually labeled "Preview"): Font size, line height
- These two sections must be visually separated (e.g., a subtle vertical divider or distinct group labels)

**Edit Only mode:**
- Editor section only: Font size, line height, show line numbers toggle, word wrap toggle, linting toggle, linter selection dropdown

The **Font Family** setting is deliberately excluded from this toolbar. It remains in the dedicated Settings page only.

All controls in this pop-down toolbar must have tooltips. Changes made in this toolbar must update the same configuration keys used by the Settings page (they are not independent copies). Changes should apply immediately (no save/apply button).

### Task 2.4 — Spec Update (Phase 2)

Update `shruggie-md-spec.md`:

- §5.2: Update to clarify this is now "View mode" (rename from "Full-View Mode").
- §5.3: Update to clarify this is now "Edit mode" (rename from "Split-View Mode").
- Add new §5.3.1 or §5.4 (renumber as needed): "Edit Only Mode" describing the full-window editor.
- §4.6 Toolbar: Update to describe the segmented mode switcher with word + icon buttons.
- Add description of the pop-down toolbar (collapsible quick-settings panel) to §5 or §4.6.
- §10.1.1: Update `lastViewMode` to include `"edit-only"` as a valid value. Add `general.editorToolbarExpanded` setting.

## Phase 3: Storage Architecture

**Dependencies:** Phase 2 complete.
**Parallelizable:** Tasks 3.1 and 3.2 are sequential. Task 3.3 can begin after 3.1.
**Estimated scope:** Large. Backend and data layer changes. This is the highest-risk phase.

### Task 3.1 — Storage Abstraction Layer

Design and implement a storage abstraction that replaces the current JSON config persistence. The abstraction must support two backends:

- **Desktop (Tauri):** SQLite via `tauri-plugin-sql`. Add the dependency: Rust crate `tauri-plugin-sql` with the `sqlite` feature, and the corresponding npm package `@tauri-apps/plugin-sql`.
- **Chrome Extension / PWA:** IndexedDB with an equivalent logical schema. The Chrome Extension currently uses `chrome.storage.sync` for config; migrate this to IndexedDB for schema parity (retain `chrome.storage.sync` only for cross-device sync of a small config subset if desired, but the primary store must be IndexedDB).

The abstraction layer lives in `src/platform/storage.ts` (or a new `src/storage/` directory). It exposes a `StorageAdapter` interface with methods for:

- Config read/write (key-value, replaces current JSON config)
- Document records (CRUD)
- Workspace records (CRUD)
- Log entries (append, query with filters)
- Edit history (append, prune by threshold)

Each backend implements this interface. The platform detection logic (already in `src/platform/`) selects the correct backend at startup.

### Task 3.2 — SQLite Schema (Desktop)

Define the SQLite schema. The database file lives in the platform-standard app data directory (§10.5) with filename `shruggie-md.db`.

Tables:

**`config`** — Application configuration (replaces JSON config file).

| Column | Type | Notes |
|--------|------|-------|
| `key` | TEXT PRIMARY KEY | Dotted path (e.g., `appearance.colorMode`, `editor.fontSize`) |
| `value` | TEXT | JSON-encoded value |
| `updated_at` | TEXT | ISO 8601 timestamp |

**`documents`** — Document registry.

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PRIMARY KEY | UUID v4 |
| `title` | TEXT | Display name (derived from filename or H1 header) |
| `source_type` | TEXT | `local`, `remote`, `internal` |
| `source_path` | TEXT NULLABLE | Absolute filesystem path (local/internal) |
| `source_url` | TEXT NULLABLE | Original URL (remote) |
| `workspace_id` | TEXT | FK to `workspaces.id` |
| `created_at` | TEXT | ISO 8601 |
| `updated_at` | TEXT | ISO 8601 |

**`workspaces`** — Workspace registry.

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PRIMARY KEY | UUID v4 |
| `name` | TEXT UNIQUE | User-facing display name |
| `type` | TEXT | `internal`, `external` |
| `path` | TEXT | Absolute path to workspace directory |
| `is_default` | INTEGER | 1 for the default workspace, 0 otherwise. Exactly one row may have `is_default = 1`. |
| `created_at` | TEXT | ISO 8601 |

**`edit_history`** — Document edit snapshots.

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | |
| `document_id` | TEXT | FK to `documents.id` |
| `content_snapshot` | TEXT | Full document content at snapshot time |
| `created_at` | TEXT | ISO 8601 |

Pruning rule: retain the most recent N snapshots per document (configurable, default 50). Pruning runs on application startup and after each save.

**`logs`** — Application log entries.

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER PRIMARY KEY AUTOINCREMENT | |
| `level` | TEXT | `debug`, `info`, `warning`, `error` |
| `source` | TEXT | Component or module name |
| `message` | TEXT | Log message |
| `metadata` | TEXT NULLABLE | JSON-encoded additional data |

**`tags`** — Tag definitions (stub, not wired to UI in this milestone).

| Column | Type | Notes |

| `id` | TEXT PRIMARY KEY | UUID v4 |

| `created_at` | TEXT | ISO 8601 |

**`document_tags`** — Many-to-many relationship (stub).

| Column | Type | Notes |
|--------|------|-------|
| `document_id` | TEXT | FK to `documents.id` |
| `tag_id` | TEXT | FK to `tags.id` |
| PRIMARY KEY | | `(document_id, tag_id)` |

### Task 3.3 — IndexedDB Schema (Chrome Extension / PWA)

Implement the equivalent schema in IndexedDB. Use the same logical table names as object store names. IndexedDB does not enforce foreign keys, so document the relationships in code comments and enforce them in the application layer.

The IndexedDB database name is `shruggie-md`. Version starts at 1. Use the standard `onupgradeneeded` migration pattern so future schema changes can increment the version.

### Task 3.4 — Config Migration

On first launch after the storage migration, the application must detect the existence of the old JSON config file and migrate its contents into the new `config` table/store. After successful migration, rename (do not delete) the old config file to `config.json.bak` as a safety net. Log a `info`-level message recording the migration.

If no old config file exists (fresh install), populate the `config` table with compiled defaults.

### Task 3.5 — Logging System

Implement a lightweight application logging system that writes to the `logs` table/store.

Add a new setting:

| Setting | Type | Default | Options | Description |
|---------|------|---------|---------|-------------|
| Log verbosity | enum | `warning` | `debug`, `info`, `warning`, `error` | Controls the minimum severity level for log entries that are persisted. Events below this threshold are discarded. |

This setting appears in a new "Advanced" section at the bottom of the Settings page. The logger is initialized early in the application startup sequence (immediately after the storage backend is ready). All existing `console.log`/`console.warn`/`console.error` calls in the codebase should be reviewed. Replace calls that represent meaningful application events (not temporary debug output) with the new logger. The logger also writes to `console` in development mode (`import.meta.env.DEV`) regardless of the verbosity setting, so development debugging is not impacted.

### Task 3.6 — Window State Persistence (Desktop Only)

On desktop, track the application window's size and position. On close, write the current dimensions and position to the `config` table. On launch, restore the window to its previous size and position. Use Tauri's window APIs to get/set these values.

Config keys: `window.width`, `window.height`, `window.x`, `window.y`, `window.maximized`.

If the stored position would place the window off-screen (e.g., the user disconnected a monitor), fall back to centered placement at the default size.

- Add new §10.6 (or appropriate subsection): "Logging" describing the logging system, verbosity setting, and log table.
- Add new §10.7 (or appropriate subsection): "Window State Persistence" describing the desktop-only window position/size restore behavior.

- §13: Add "Tags" to future considerations. Describe the stub tables and the intent for a future tags browsing window.

## Phase 4: Workspace System

**Dependencies:** Phase 3 complete (storage layer must be in place).
**Parallelizable:** Tasks 4.1 through 4.4 are sequential. Task 4.5 (Spec) can be drafted in parallel with 4.4.
**Estimated scope:** Large. New UI panels, new data model, terminology migration.

### Task 4.1 — Terminology Migration

Replace all occurrences of "Library," "Mount," and "Mounted directory" throughout the codebase and documentation with "Workspace" terminology.

- UI strings: "Library" becomes "Workspaces" (plural when referring to the navigation target/view).
- Config keys: any key containing `library` is renamed. Maintain a migration mapping so old config keys are picked up during config migration (from Phase 3's migration logic, extend it to handle the rename).
- Component directories: rename `src/components/Library/` to `src/components/Workspaces/`.
- Spec terminology table (§1.3): replace the "Library" and "Mount" entries with "Workspace" definitions.
- File names, imports, and test files: update all references.

This is a sweeping rename. Use a methodical search-and-replace approach. Compile and run tests after the rename to catch any missed references.

### Task 4.2 — Default Workspace

Create the concept of a default workspace:

- On first launch (or when no workspaces exist in the database), the application creates a default workspace.
- The default workspace's `type` is `internal` and its `path` points to `{app_data_dir}/workspaces/Default/`.
- The default workspace cannot be deleted or renamed by the user.
- Its `is_default` flag is `1` in the database.
- The directory is created on disk if it does not exist.

On Chrome Extension / PWA: the default workspace is a virtual workspace backed by IndexedDB storage. Its `path` field stores a logical identifier (e.g., `__internal__/Default`) rather than a filesystem path. Document storage on these platforms is IndexedDB blob storage keyed by document ID.

### Task 4.3 — Workspace CRUD

Implement workspace management:

**Creating a workspace:**
- The user clicks "Add Workspace" at the bottom of the workspace list.
- A dialog appears with two options:
  1. **Internal:** Enter a name only. The workspace directory is created as a subfolder of `{app_data_dir}/workspaces/{name}/`. Enforce name uniqueness (case-insensitive comparison). Validate against OS-disallowed filename characters (on Windows: `< > : " / \ | ? *`, null bytes, reserved names like `CON`, `PRN`, etc.; on macOS/Linux: `/` and null bytes). Display a clear error message if validation fails.
  2. **External:** Enter a name and select a directory path (via native directory picker on desktop). The directory must already exist. The path is stored as-is. Name uniqueness is enforced independently of the path.
- On Chrome/PWA: only the "Internal" option is available (no filesystem access). The directory picker is hidden or disabled with an explanatory note.

**Workspace list panel:**
- Displayed as a collapsible right-aligned panel on the Workspaces view.
- Lists all workspaces, starting with "Default" at the top (always first, visually distinguished, e.g., a subtle badge or pin icon).
- Each workspace row shows the workspace name and an ellipsis (`...`) button at the right edge.
- Clicking the ellipsis opens a settings modal for that workspace (see Task 4.4).
- "Add Workspace" button at the bottom of the list.

**Workspace content area:**
- When a workspace is selected, the main content area displays the files in that workspace in a sortable, filterable data table (carrying forward the current library table design from §9.3).
- The table columns remain: Title, Path, Last Edited, Created.
- File extension filtering, recursion toggle, and hidden file toggle are now per-workspace settings (see Task 4.4).

### Task 4.4 — Per-Workspace Settings

Each workspace has its own settings, accessible via the ellipsis button on the workspace list panel. These settings open in a modal dialog.

Per-workspace settings (carried forward from the current Library settings in §9):

| Setting | Type | Default | Notes |
|---------|------|---------|-------|
| Recursive traversal | boolean | `true` | Whether to scan subdirectories |
| Show hidden files | boolean | `false` | Whether to include dotfiles |
| Independent extension rules | boolean | `false` | When enabled, this workspace uses its own extension whitelist instead of the global one |
| Extension whitelist | string[] | (global default) | Only visible when independent extension rules is enabled |

For **external** workspaces, an additional read-only field shows the mapped directory path.

For the **default** workspace, the delete button is absent or disabled. The name field is read-only.

For other workspaces, the modal includes a "Delete Workspace" button (with confirmation dialog). Deleting an internal workspace removes the directory and its contents from disk (with a warning). Deleting an external workspace only removes the workspace record from the database; the directory on disk is not touched.

Store per-workspace settings in the `workspaces` table as an additional `settings` TEXT column containing JSON, or in a separate `workspace_settings` table. Choose whichever approach is cleaner for the storage abstraction. If adding a column, update the schema from Phase 3 accordingly (this is acceptable since we have not released Phase 3 yet).

The Settings page currently has a "Library" section (§5.5, item 6). Remove it entirely. All of its functionality has moved to per-workspace settings in the workspace panel. The Settings page should no longer reference library, mount, or workspace directory configuration.

## Phase 5: I/O Overhaul

**Dependencies:** Phase 4 complete (workspace system must be in place).
**Parallelizable:** Tasks 5.1 and 5.2 can be done in parallel. Task 5.3 depends on 5.1. Task 5.4 depends on 5.3.
**Estimated scope:** Medium-Large. Touches file entry, save, and export flows.

### Task 5.1 — Unified Open Dialog

Replace all file entry points with a single "Open" dialog. This dialog is invoked by:

- The toolbar "Open" button (or `Ctrl/Cmd+O`).
- CLI invocation (`shruggie-md <filepath>` or `shruggie-md --url <url>`). These still work, but internally route through the same Open logic.
- OS file association (double-click a `.md` file). Also routes through the Open logic.

The Open dialog contains:

1. **Source selector:** Two tabs or a toggle: "File" and "URL."
   - File tab: A native file picker button (desktop) or file input (Chrome/PWA). Filters to recognized markdown extensions.
   - URL tab: A text input for a remote URL (carries forward the current URL input behavior).
2. **"Open in" workspace selector:** A dropdown listing all workspaces. Defaults to the default workspace. The final option in the dropdown is "Add a new workspace..." which, when selected, expands the dialog to include the workspace creation fields (name, type selection, optional directory picker) inline.
3. **Optional file name:** A text input for overriding the document name. Not required. If left blank:
   - For local files: the filename (without extension) is used.
   - For remote files: attempt to extract the H1 header from the fetched content. If no H1 is found, attempt to derive a name from the URL's last path segment. If neither works, generate a timestamp-based name.

**Behavior on "Open":**
- Local file: If the file is already inside the selected workspace's directory, open it directly. If the file is outside the workspace, copy it into the workspace directory, then open the copy. Register the document in the `documents` table.
- Remote file: Fetch the content, save it into the selected workspace directory, register it in the `documents` table with `source_type = 'remote'` and `source_url` set. Open the saved copy.

**CLI and OS association routing:** When the app launches with a file path argument or via OS file association, skip showing the dialog and instead silently open the file into the default workspace (copying it in if it is not already inside the workspace directory). The user can later move it to a different workspace via workspace management.

### Task 5.2 — Save Behavior Update

The current Save/Save As behavior (from v0.2.0) is mostly correct but needs to integrate with the workspace and document model:

- **Save** (`Ctrl/Cmd+S`): Writes the current editor content to the document's `source_path`. Updates the `documents` table `updated_at` timestamp. Creates an edit history snapshot.
- **Save As** (`Ctrl/Cmd+Shift+S`): Opens a native save dialog. If the user picks a location inside an existing workspace, the document record updates its `source_path`. If the user picks a location outside all workspaces, the document is saved there (no workspace association change, the document retains its original workspace). Update the `documents` table accordingly.
- **New Document from Editor:** The editor should support creating a new blank document. Add a toolbar action (or include it in the Open dialog) that creates a new empty document, prompts for a workspace selection, and opens the editor. The new document is registered in the `documents` table with `source_type = 'internal'`.

### Task 5.3 — Export System

Replace the current "Converting" language (§6.3) with an "Export" system.

- **Export** (`Ctrl/Cmd+Shift+E` or toolbar button): Opens an export dialog.
- The dialog offers format selection: HTML, PDF, Markdown (`.md`).
- Selecting a format and confirming opens a native OS file picker (on all platforms where available) for the user to choose the destination filename and location. This is one of the few times the user interacts with an OS-level file picker outside of workspace management.
- HTML export: self-contained file with inlined styles (current behavior, carried forward).
- PDF export: current print-pathway behavior (§6.3), carried forward.
- Markdown export: writes the raw editor content to the chosen destination. Useful for exporting a copy outside of workspaces.

Update keyboard shortcuts:
- `Ctrl/Cmd+Shift+H` (HTML export) and `Ctrl/Cmd+Shift+P` (PDF export) should still work as direct shortcuts that skip the format selection dialog and go straight to the file picker for their respective formats.
- `Ctrl/Cmd+Shift+E` opens the format selection dialog.
