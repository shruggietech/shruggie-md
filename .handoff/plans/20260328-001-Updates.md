## Summary

This plan closes a set of gaps identified during a build-process audit. The core issue: while `tauri.conf.json` already runs `pnpm build` (which invokes Vite, which clears `dist/` by default) via `beforeBuildCommand`, this guard only works when the build is invoked through the Tauri CLI. An agent or contributor who bypasses that path (running `cargo build` directly, or encountering a silent frontend build failure with an old `dist/` still on disk) will silently package stale frontend output.

The fix is defense-in-depth: explicit `dist/` deletion at the npm script level so the guard works regardless of how the build is invoked, plus documentation updates to make this the only recommended path.

Four deliverables:

1. **Build scripts:** Add explicit `dist/` deletion to `tauri:build` and `tauri:dev` in `package.json`, via a small cross-platform helper script.
2. **README.md:** Add a build hygiene note to the contributor setup sections.
3. **RELEASING.md:** Insert a clean-build verification step before tagging.
4. **Closure report:** Write `20260327-002-Report.md` to formally close the remedial sprint plan that was never reported on.

## Tasks

### Task 1: Create `scripts/clean-dist.js`

Create a minimal cross-platform script that deletes only `dist/` (not the full clean). This avoids platform-specific shell syntax in `package.json` and avoids adding a dependency like `rimraf`.

**File:** `scripts/clean-dist.js`

**Contents:**

```js
import { rmSync } from 'fs';

rmSync('dist', { recursive: true, force: true });
```

This works cross-platform because the project is `"type": "module"` and targets Node.js 22+. The `force: true` option means it silently succeeds if `dist/` does not exist.

### Task 2: Update `package.json` scripts

Replace the current thin passthroughs with dist-cleaning variants:

**Before:**

```json
"tauri:dev": "tauri dev",
"tauri:build": "tauri build",
```

**After:**

```json
"tauri:dev": "node scripts/clean-dist.js && tauri dev",
"tauri:build": "node scripts/clean-dist.js && tauri build",
```

**Rationale:** This ensures `dist/` is always deleted before invoking the Tauri CLI, regardless of whether `beforeBuildCommand`/`beforeDevCommand` succeeds or fails. If the frontend build (triggered by Tauri's before-commands) fails, there will be no `dist/` to fall back to, and Tauri will fail with a clear error instead of silently packaging stale output.

**Verification:** Run `pnpm tauri:build` (or `pnpm tauri:dev` if a Rust toolchain is available) and confirm the build completes successfully. The dist-clean step should appear in the console output before the Tauri CLI starts.

## Notes

- The `beforeBuildCommand` and `beforeDevCommand` in `tauri.conf.json` should NOT be modified. They are working correctly. The `clean-dist.js` step is an additional layer of defense at the npm script level, not a replacement for Tauri's built-in before-commands.
- The CI/CD workflows (`ci.yml`, `release.yml`) do not need changes. The release workflow uses `tauri-apps/tauri-action@v0`, which invokes the Tauri CLI directly (bypassing npm scripts), but it always starts from a fresh checkout with no pre-existing `dist/`. The stale-dist problem only affects local development and agent sessions where `dist/` may persist from a previous build.
