## Problem Statement

The v0.3.0 milestone code is committed and pushed. The source compiles and runs correctly via `pnpm tauri dev`. However, the release binaries produced by the build agent contain **stale v0.2.0 frontend output** embedded in a v0.3.0-versioned Tauri shell. The agent ran `pnpm tauri build` against a cached `dist/` directory that predated the v0.3.0 frontend changes.

The root cause is a missing step in the build sequence: the agent did not delete and regenerate `dist/` before invoking the Tauri build. Tauri embeds the contents of `dist/` at compile time. If `dist/` is stale, the binary is stale regardless of what the source tree contains.

This is the third instance of an agent shipping unverified artifacts:

- v0.1.0: Agent removed `bundle.windows` from `tauri.conf.json`, never smoke-tested the installed binary.
- v0.1.1: Hotfix worked in dev mode but production NSIS build was not executed by the agent.
- v0.3.0: Agent wrote real code but built binaries from cached frontend output.

The pattern is consistent: agents pass typecheck and unit tests, then declare success without launching or inspecting the built artifact. This plan addresses both the immediate build failure and the systemic governance gap.

## Objectives

1. Create `CLAUDE.md` at the repo root with persistent standing rules that every agent session reads before executing any work.
2. Produce verified v0.3.0 release binaries from a clean build.
3. Remove orphaned dead code flagged in the v0.3.0 after-action report.
4. Commit all changes and push. Do not tag a release.

## Standing Rules

All standing rules from `20260327-001-Updates.md` remain in effect. This plan adds one:

7. **Clean frontend build before Tauri build.** Before running `pnpm tauri build` or `pnpm tauri dev`, the agent must delete the `dist/` directory and run `pnpm run build` to regenerate it from the current source tree. Building Tauri against a pre-existing `dist/` is prohibited.

## Sprint Tasks

Tasks are numbered and must be completed in order.

### Task 1 — Create `CLAUDE.md`

Create the file `CLAUDE.md` at the repository root. This file is read by OpenCode CLI (and compatible tools) at the start of every agent session. It must contain all standing rules that apply across sprints, so that future sprint plans can reference them by number rather than restating them.

Write the file with the following content structure. Do not add anything beyond what is specified here.

**Section 1: Project Identity**

- Project name, repo URL, license.
- Tech stack summary (one line each for frontend, backend, build tooling).
- Spec file location: `shruggie-md-spec.md` at repo root.

**Section 2: Standing Rules**

Number each rule. Include all seven rules accumulated across prior sprint plans:

1. No release tagging. The agent must not create git tags or GitHub releases. That decision is reserved for the project owner after manual smoke testing.
2. Verify before declaring done. Every sprint ends with a verification task. The agent must run the application (`pnpm tauri dev` at minimum) and visually confirm the changes work. Typecheck and test-only verification is insufficient.
3. Spec fidelity. When a task references a spec section, read that section before implementing. Do not rely on cached knowledge or assumptions about spec content.
4. No scope creep. Complete only the tasks listed in the current sprint plan. Do not improve adjacent systems or refactor code outside the plan scope.
5. Existing tests must pass. At the end of every sprint, `pnpm run test` must report zero failures. If a change breaks existing tests, fix them within the same sprint.
6. CHANGELOG accuracy. Update `CHANGELOG.md` with an `[Unreleased]` section reflecting actual changes made. Do not fabricate test counts, feature completions, or metrics.
7. Clean frontend build before Tauri build. Before running any Tauri build command (`pnpm tauri build`, `pnpm tauri dev`, or equivalent), delete the `dist/` directory and run `pnpm run build`. Building Tauri against a pre-existing `dist/` is prohibited. The correct sequence is always: `rm -rf dist && pnpm run build && pnpm tauri build`.

**Section 3: Build Procedure**

Document the mandatory build sequence for producing release artifacts:

```
1. pnpm install
2. pnpm run typecheck
3. pnpm run test
4. Remove dist/ entirely
5. pnpm run build
6. pnpm tauri build
7. Launch the built artifact (not dev mode) and visually verify
```

State explicitly: steps 4 and 5 are not optional. Skipping them produces binaries with stale frontend code.

**Section 4: File Conventions**

- Sprint plans: `.handoff/plans/YYYYMMDD-NNN-Updates.md`
- After-action reports: `.handoff/reports/YYYYMMDD-NNN-Report.md`
- Sprint plans and reports are AI-to-AI documents. Use pure Markdown with no embedded HTML.
- Brand assets: `assets/brand/` directory. Filenames use the exact format `Shruggie Tech2-01.png` through `Shruggie Tech2-06.png`.
- The pre-made `.ico` file is used directly; do not regenerate it.

**Section 5: Known Agent Failure Modes**

List these explicitly so future agents are primed to avoid them:

- Building Tauri without first regenerating `dist/` from current source.
- Declaring UI changes complete based on typecheck/test without visual verification.
- Reporting file paths in after-action reports that do not match actual file locations.
- Removing or overwriting `tauri.conf.json` configuration blocks (especially `bundle.windows`) without understanding the downstream effects.
- Fabricating completion status, test counts, or metrics in after-action reports.
- Making release decisions (tagging, publishing) that are reserved for the project owner.

**Format requirements:** Pure Markdown. No HTML tags. No anchors, no page breaks, no inline styles. This is an AI-only document.

### Task 2 — Clean Frontend Build

Execute the following sequence exactly:

```
Remove-Item -Recurse -Force dist -ErrorAction SilentlyContinue
Remove-Item -Recurse -Force src-tauri/target/release/bundle -ErrorAction SilentlyContinue
pnpm install
pnpm run build
```

After `pnpm run build` completes, verify the output:

- [ ] `dist/` directory exists and contains freshly generated files.
- [ ] `dist/index.html` exists.
- [ ] `dist/assets/` contains `.js` and `.css` files with hashes in their names.
- [ ] No files in `dist/assets/` have timestamps older than the current session.

### Task 3 — Tauri Release Build

Run:

```
pnpm tauri build --bundles nsis
pnpm tauri build --bundles msi
```

If `--bundles nsis` fails with the DLL hashing issue documented in the v0.3.0 report, delete `src-tauri/target/release/bundle/nsis/` and retry.

After both builds complete, verify the output:

- [ ] NSIS installer exists at `src-tauri/target/release/bundle/nsis/Shruggie Markdown_0.3.0_x64-setup.exe`.
- [ ] MSI installer exists at `src-tauri/target/release/bundle/msi/Shruggie Markdown_0.3.0_x64_en-US.msi`.
- [ ] Both files have timestamps from the current session.

### Task 4 — Artifact Smoke Test

Install the NSIS artifact on the local Windows machine. After installation, launch the application from the Start Menu or install directory (not via `pnpm tauri dev`).

Verify visually:

- [ ] The application window opens without errors.
- [ ] Scrollbars are thin and themed (not default browser scrollbars).
- [ ] The toolbar shows the five-mode segmented control (View, Edit, Edit Only, Workspaces, Settings).
- [ ] Clicking the chevron toggle opens/closes the EditorToolbarPanel below the toolbar.
- [ ] The Workspaces view renders (not "Library").
- [ ] Settings page shows six sections: Appearance, Editor, Preview, Markdown Engine, File Extensions, Advanced.
- [ ] Ctrl+O opens the Open dialog with File/URL tabs.
- [ ] Ctrl+Shift+E opens the Export dialog with HTML/PDF/Markdown options.
- [ ] The title bar or window title shows "Shruggie Markdown" (not a version mismatch).

If any of these checks fail, stop and diagnose before proceeding.

### Task 5 — Dead Code Cleanup

Remove orphaned files identified in the v0.3.0 after-action report:

1. Delete `src/components/Toolbar/UrlInput.tsx` if it exists and is not imported anywhere.
2. Delete `src/hooks/useRemoteFetch.ts` if it exists and is not imported anywhere.

Before deleting each file, run a search to confirm no active imports reference it:

```
Select-String -Recurse -Path "src/" -Pattern "UrlInput" -Include "*.ts","*.tsx"
Select-String -Recurse -Path "src/" -Pattern "useRemoteFetch" -Include "*.ts","*.tsx"
```

If any active imports are found, do not delete the file. Report the finding instead.

After deletion, run:

- [ ] `pnpm run typecheck` reports zero errors.
- [ ] `pnpm run test` reports zero failures.

## Deferred: Company Knowledge Base

The standing rules in `CLAUDE.md` are project-specific but encode principles that apply to every ShruggieTech project that uses AI build agents:

- Agents must launch and visually verify built artifacts before declaring completion.
- Agents must not make release decisions.
- Agent-produced metrics and completion claims require independent verification.
- Build pipelines with embedded assets (Tauri, Electron, etc.) require explicit cache invalidation before builds.

These should be elevated to a company-wide AI agent governance document in the ShruggieTech knowledge base. That document would be referenced by all project-level `CLAUDE.md` files rather than restating the rules inline. This is out of scope for the current sprint but should be prioritized before the next new project.
