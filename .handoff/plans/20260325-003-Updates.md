## Problem Statement

The v0.1.0 Windows release is non-functional. The NSIS installer (`.exe`) completes without error, but the application does not launch, either from the post-install "run" checkbox or from the desktop shortcut. The MSI installer has not been independently verified. The install path defaults to `%LOCALAPPDATA%\Shruggie Markdown\`, which violates the project's directory naming convention.

These problems were not caught during the v0.1.0 sprint because no manual smoke test was performed. CI verification confirmed only that the project compiles and that tests pass, not that the packaged application starts.

## Objectives

1. Diagnose and fix the application launch failure on Windows.
2. Correct the Windows install path to `%LOCALAPPDATA%\shruggie-tech\shruggie-md\`.
3. Fix the remaining low-effort known issues from the prior report.
4. Update `shruggie-md-spec.md` §10.5 to reflect the correct install path convention.
5. Verify the fix by actually launching the application, not just by confirming a clean build.

## Task List

Complete these tasks in order.

### Task 1 — Diagnose the launch failure

Using the investigation checklist above, determine why the application does not start. The most likely root causes (in order of probability) are:

1. **`frontendDist` misconfiguration.** If `tauri.conf.json` has `build.frontendDist` set to a path that does not match the Vite output directory, the packaged app will contain no frontend assets. The Tauri window will either not open or will open blank/white. Verify that `frontendDist` is set to `"../dist"` (relative to `src-tauri/`).
2. **Tauri entry point crash.** The prior sprint noted an unused `use tauri::Manager;` import in `lib.rs` and a possibly-unnecessary `features = ["unstable"]` flag. Review the Tauri entry point for panics, unwrap calls on None/Err values, or logic that conditionally skips window creation.
3. **NSIS shortcut target mismatch.** The CI fix cycle (commit `ecbcf6b`) removed the entire `bundle.windows` block from `tauri.conf.json`. If this block previously contained configuration that the NSIS bundler relies on to set the correct executable path in shortcuts, the shortcut may point to a nonexistent binary. Compare the shortcut target path with the actual location of the `.exe` inside the install directory.
4. **WebView2 runtime missing.** Tauri 2 on Windows requires the Edge WebView2 runtime. If the NSIS installer does not bundle or bootstrap the WebView2 installer, the app will silently fail to launch on systems that do not have it. Check whether `tauri.conf.json` includes `bundle.windows.webviewInstallMode` or equivalent. If not, add it with `"downloadBootstrapper"` to ensure WebView2 is available.

Do not guess. Reproduce the failure locally, identify the exact cause, and document it before proceeding to a fix. If you cannot reproduce locally (e.g., you are not on Windows), run `pnpm tauri dev` on whatever platform you have and check that the window opens and renders the frontend. If the dev build works but the production build does not, the problem is in the bundler/installer configuration, not the application code.

### Task 2 — Fix the launch failure

Apply the fix corresponding to whatever root cause you identified in Task 1. If multiple causes are present, fix all of them.

### Task 3 — Fix the install path

The NSIS installer must install to `%LOCALAPPDATA%\shruggie-tech\shruggie-md\`, not `%LOCALAPPDATA%\Shruggie Markdown\`.

In `tauri.conf.json`, add or update the `bundle.windows.nsis` object:

```json
"windows": {
  "nsis": {
    "installMode": "currentUser",
    "installerIcon": "icons/icon.ico",
    "displayLanguageSelector": false
  },
  "webviewInstallMode": {
    "type": "downloadBootstrapper"
  }
}
```

The install directory itself is derived from the Tauri `identifier` field. Tauri 2 NSIS uses the identifier to construct the install path. Verify that the `identifier` field follows reverse-domain notation (e.g., `com.shruggie-tech.shruggie-md` or `tech.shruggie.shruggie-md`). If it does not, correct it. The resulting default install path under `%LOCALAPPDATA%` should resolve to `shruggie-tech\shruggie-md`.

If Tauri's default NSIS path derivation does not produce the correct directory structure from the identifier alone, consult the Tauri 2 NSIS documentation (check `node_modules/@tauri-apps/cli/config.schema.json` for the `NsisConfig` schema, as done in the prior sprint) and set the appropriate field to override the install directory explicitly.

### Task 5 — Update the spec

In `shruggie-md-spec.md`, update the §10.5 Storage Locations table. The Windows path must change from:

```
%LOCALAPPDATA%\shruggie-md\
```

to:

```
%LOCALAPPDATA%\shruggie-tech\shruggie-md\
```

This aligns the spec with the company's `{company-slug}\{product-slug}` directory convention used across other ShruggieTech projects.

Do not change the macOS, Linux, Chrome, or PWA paths.
