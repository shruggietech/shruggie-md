# Shruggie Markdown - Product Brainstorm

## Summary

Shruggie Markdown is a cross-platform markdown viewer, editor, and converter application. It targets desktop (Windows 11, macOS, Linux), browser (Chrome extension, possibly Chrome OS), and mobile (Android, possibly via PWA). The application provides a modern, themeable UI with file management capabilities when installed locally.

## Core Capabilities

### Viewing

- Render markdown files with a modern, themeable UI.
- Local files: support live updates when the source file changes on disk.
- Remote files: fetch and render raw markdown content from a URL. Provide usage instructions via a modal info popup triggered by a small info badge in the UI.

### Editing

- Toggle between full-view (preview only) and split-view (editor + preview) with a single button click.
- Optional markdown linting with actionable tooltip popups when issues are detected.

### Converting

- Markdown to HTML.
- Markdown to PDF:
  - The conversion pipeline is MD to HTML to print-ready output with page numbers and no URL footers.
  - Auto-format feature: replace `<hr>` elements with print-friendly page-break markup (reference the "Markdown to PDF Converter" web app for the specific approach).

## Configuration and Preferences

### Theme and Appearance

- Light, Dark, and System color modes. Default to dark mode.
- A small collection of additional visual styles (reference OpenCode's UI for tone and scope).
- Font preferences with a similar option set to OpenCode's settings.

### Technical Options

- Markdown compilation engine selection (for advanced users; reference the "Markdown Viewer" Chrome plugin for inspiration on what to expose).

### File Extension Rules

- A configurable set of file extensions the application will attempt to read as markdown.
- Files that do not match the configured extension pattern(s) are ignored. Display an appropriate error message suggesting the user modify this setting.
- These rules apply in simple file viewing mode. Library mode has a separate toggle to either adopt these same rules or define its own set of extensions (see Library Mode below).

### Storage

- On all desktop platforms (Windows, macOS, Linux), preferences and configuration live in the user's standard application data storage location.

## Library Mode (Local Install Only)

When installed as a native desktop application (not used as a PWA), the app provides a flat file management interface.

### Setup

- The user configures a single directory to be "mounted" as a markdown library via settings.

### Behavior

- Track library contents in a table with the following columns:
  - **Title**: file name without the file extension.
  - **Path**: absolute path to the file.
  - **Last Edited**: ISO 8601 date-time with UTC offset of the file's last modification.
  - **Created**: ISO 8601 date-time with UTC offset of the file's creation.
- The library directory may contain non-markdown files; they are simply ignored.
- By default, recursively traverse into sub-directories. This is toggleable in the library section of the config settings.
- Hidden files and directories (e.g., `.git`) are ignorable via a separate toggle.

### Extension Rules in Library Mode

- A toggle determines whether library mode follows the same file extension rules used in simple file viewing mode.
- If the toggle is off, library mode exposes its own independent set of configurable file extensions.

## Release Targets

### Desktop

All desktop targets share the following requirements:

- The system PATH includes a CLI-invocable `shruggie-markdown` command that launches the application.
- Preferences stored in the platform-standard application data location.

Platform-specific notes:

- **Windows 11** (primary target):
  - Offer a desktop icon during installation.
  - Prompt the user to set the application as the default handler for `*.md` and `*.markdown` files in the Windows "open with" settings.
- **macOS**: no additional platform-specific requirements identified yet.
- **Linux**: no additional platform-specific requirements identified yet.

### Browser

- **Chrome Extension**: target the Chrome Web Store. Investigate whether the same extension can also target Chrome OS (needs further research).

### Mobile

- **Android**: investigate a PWA "install to home screen" approach (needs further research).

## Open Questions

These items were flagged in the original brainstorm as needing further research before specification:

1. Can the Chrome extension also target Chrome OS, or does Chrome OS require a separate build/packaging approach?
2. Is a PWA the right vehicle for the Android target, or should a native wrapper (e.g., Tauri, Electron, Capacitor) be evaluated?
3. What is the intended technology stack? The cross-platform requirements (desktop + browser extension + possible PWA) will heavily constrain this choice.
4. Should remote URL fetching support authenticated endpoints, or only publicly accessible raw content?
5. What markdown compilation engines should be offered as selectable options in the technical settings?
