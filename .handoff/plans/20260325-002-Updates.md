## 1. Context and Objectives

The MVP build (20260325-001) produced a functional codebase: 92 source files, 254 passing tests, zero TypeScript errors, three build targets (Web/PWA, Chrome Extension, Desktop/Tauri). All code is currently untracked in git.

This sprint plan covers the infrastructure, tooling, and polish work required to make the project release-ready. The objectives are:

1. Replace all placeholder icons with branded ShruggieTech assets.
2. Establish a changelog and formalize the release-cutting process.
3. Create developer-friendly shell scripts for common workflows.
4. Set up GitHub Actions for CI (test on push/PR) and CD (build release artifacts on version tag).
5. Complete the linter adapter stubs left from the MVP build.
6. Clean up housekeeping items (CSP, unused dependencies, license fields).
7. Expand the README from stub to a proper project landing page.
8. Commit all work and tag v0.1.0.

**SVG handling:** The existing placeholder includes an `icon.svg`. Since no vector source of the brand icon exists, replace it with a raster-embedded SVG:

```bash
printf '<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 512 512"><image width="512" height="512" href="data:image/png;base64,' > public/icons/icon.svg
base64 -w 0 assets/brand/icon-512.png >> public/icons/icon.svg
printf '"/></svg>' >> public/icons/icon.svg

Check the resulting file size. If it exceeds 100 KB, delete the SVG and remove any SVG references from `public/manifest.json` (the PNG entries are sufficient).

#### 2.2 Create RELEASING.md

Create `RELEASING.md` at the repository root. This is a human-facing procedural document.

Content:

```markdown
# Releasing Shruggie Markdown

This document describes the process for cutting a new release of Shruggie Markdown.

## Version Bump Locations

Every release requires updating the version string in these files:

| File | Field |
|------|-------|
| `package.json` | `"version"` |
| `src-tauri/tauri.conf.json` | `"version"` |
| `src-tauri/Cargo.toml` | `version` (under `[package]`) |
| `extension/manifest.json` | `"version"` |

All four files must contain the same version string. The version follows [Semantic Versioning](https://semver.org/): `MAJOR.MINOR.PATCH`.

## Release Procedure

### 1. Prepare the Release Branch

```bash
git checkout main
git pull origin main
git checkout -b release/vX.Y.Z
```

### 2. Update Version Numbers

Update the version string in all four files listed above. Use find-and-replace or the version bump script:

```bash
# PowerShell
./scripts/bump-version.ps1 -Version "X.Y.Z"

# Bash
./scripts/bump-version.sh X.Y.Z
```

### 3. Update the Changelog

Edit `CHANGELOG.md`:

1. Move items from `[Unreleased]` into a new version section: `## [X.Y.Z] - YYYY-MM-DD`
2. Create a fresh empty `## [Unreleased]` section at the top.
3. Add the comparison link at the bottom: `[X.Y.Z]: https://github.com/shruggietech/shruggie-md/compare/vPREVIOUS...vX.Y.Z`
4. Update the `[Unreleased]` link to compare against the new tag.

### 4. Commit and Tag

```bash
git add -A
git commit -m "release: vX.Y.Z"
git tag vX.Y.Z
```

### 5. Push

```bash
git push origin release/vX.Y.Z
git push origin vX.Y.Z
```

Pushing the tag triggers the GitHub Actions release workflow, which:

- Runs the full test suite.
- Builds desktop installers for Windows (MSI), macOS (DMG), and Linux (AppImage).
- Creates a GitHub Release with the built artifacts attached.
- Extracts the changelog entry for the version and uses it as the release notes.

### 6. Merge

After the release workflow completes and artifacts are verified:

```bash
git checkout main
git merge release/vX.Y.Z
git push origin main
```

### 7. Post-Release

- Verify the GitHub Release page has all expected artifacts.
- Upload the Chrome extension build (`extension/dist/`) to the Chrome Web Store (manual process).
- Deploy the PWA build (`dist/`) to the hosting provider (manual process or separate workflow).

## Tag Format

Tags use the format `vX.Y.Z` (e.g., `v0.1.0`, `v1.0.0`). The leading `v` is required; the GitHub Actions release workflow triggers on tags matching `v*`.

## Chrome Web Store and PWA Deployment

These are currently manual processes:

- **Chrome Web Store:** Zip `extension/dist/`, upload via the Chrome Web Store Developer Dashboard.
- **PWA:** Deploy `dist/` to a static hosting provider (e.g., Cloudflare Pages, Netlify, Vercel).

Automated deployment for these targets is a future consideration.
```

#### 4.2 Release Workflow: `.github/workflows/release.yml`

**Triggers:** Push of a tag matching `v*` (e.g., `v0.1.0`).

This workflow builds desktop installers for all three platforms using `tauri-apps/tauri-action`, which handles multi-platform compilation on GitHub-hosted runners. It creates a GitHub Release and attaches the built artifacts.

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  create-release:
    permissions:
      contents: write
    runs-on: ubuntu-latest
    outputs:
      release_id: ${{ steps.create_release.outputs.id }}
    steps:
      - uses: actions/checkout@v4

      - name: Extract version from tag
        id: version
        run: echo "version=${GITHUB_REF#refs/tags/v}" >> $GITHUB_OUTPUT

      - name: Extract changelog entry
        id: changelog
        run: |
          version="${{ steps.version.outputs.version }}"
          body=$(awk "/^## \\[${version}\\]/{flag=1; next} /^## \\[/{flag=0} flag" CHANGELOG.md)
          echo "body<<EOF" >> $GITHUB_OUTPUT
          echo "$body" >> $GITHUB_OUTPUT
          echo "EOF" >> $GITHUB_OUTPUT

      - name: Create GitHub Release
        id: create_release
        uses: softprops/action-gh-release@v2
        with:
          tag_name: ${{ github.ref_name }}
          name: Shruggie Markdown ${{ github.ref_name }}
          body: ${{ steps.changelog.outputs.body }}
          draft: true
          prerelease: ${{ contains(steps.version.outputs.version, '-') }}

  build-tauri:
    needs: create-release
    permissions:
      contents: write
    strategy:
      fail-fast: false
      matrix:
        include:
          - platform: windows-latest
            args: ''
          - platform: macos-latest
            args: --target aarch64-apple-darwin
          - platform: ubuntu-22.04
            args: ''
    runs-on: ${{ matrix.platform }}
    steps:
      - uses: actions/checkout@v4

      - name: Install Linux dependencies
        if: matrix.platform == 'ubuntu-22.04'
        run: |
          sudo apt-get update
          sudo apt-get install -y libwebkit2gtk-4.1-dev libappindicator3-dev librsvg2-dev patchelf

      - name: Setup pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 9

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm

      - name: Install Rust (stable)
        uses: dtolnay/rust-toolchain@stable
        with:
          targets: ${{ matrix.platform == 'macos-latest' && 'aarch64-apple-darwin' || '' }}

      - name: Rust cache
        uses: swatinem/rust-cache@v2
        with:
          workspaces: src-tauri -> target

      - name: Install frontend dependencies
        run: pnpm install --frozen-lockfile

      - name: Build Tauri app
        uses: tauri-apps/tauri-action@v0
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          releaseId: ${{ needs.create-release.outputs.release_id }}
          args: ${{ matrix.args }}

  build-extension:
    needs: create-release
    permissions:
      contents: write
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 9

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Build extension
        run: pnpm run build:extension

      - name: Zip extension
        run: cd extension/dist && zip -r ../../shruggie-md-chrome-extension.zip .

      - name: Upload extension to release
        uses: softprops/action-gh-release@v2
        with:
          tag_name: ${{ github.ref_name }}
          files: shruggie-md-chrome-extension.zip

  publish-release:
    needs: [build-tauri, build-extension]
    permissions:
      contents: write
    runs-on: ubuntu-latest
    steps:
      - name: Publish release (remove draft status)
        uses: actions/github-script@v7
        with:
          script: |
            const releases = await github.rest.repos.listReleases({
              owner: context.repo.owner,
              repo: context.repo.repo,
            });
            const draft = releases.data.find(r => r.draft && r.tag_name === '${{ github.ref_name }}');
            if (draft) {
              await github.rest.repos.updateRelease({
                owner: context.repo.owner,
                repo: context.repo.repo,
                release_id: draft.id,
                draft: false
              });
            }
```

**Agent notes on the release workflow:**

- The release is created as a draft first. Tauri action and the extension build job upload artifacts to the draft. After all builds succeed, the final job publishes it (removes draft status). If any build fails, the draft remains unpublished.

#### 5.2 Implement markdownlint Adapter

File: `src/linters/markdownlint-adapter.ts`

The adapter already implements the `LinterAdapter` interface and has a `lint(source: string): LintDiagnostic[]` method that currently returns `[]`.

Implementation:

1. Import `markdownlint` (the synchronous `markdownlint.sync` API).
2. In the `lint()` method, call `markdownlint.sync({ strings: { content: source } })`.
3. Map the result array to `LintDiagnostic[]`:
   - `markdownlint` results have `lineNumber`, `ruleNames`, `ruleDescription`, `errorRange` (nullable `[column, length]`).
   - Convert `lineNumber` (1-indexed) to a character offset by summing the lengths of all preceding lines in the source string.
   - If `errorRange` is present, use `errorRange[0]` (1-indexed column) to compute the precise character offset and `errorRange[1]` for the length.
   - If `errorRange` is null, mark the entire line.
   - Map severity: markdownlint does not distinguish error/warning natively. Treat all as `"warning"`.
   - Set `ruleId` to the first rule name (e.g., `"MD001"`), `message` to `ruleDescription`.
4. Wrap the call in a try-catch. On error, log to console and return `[]`.

#### 5.3 Implement remark-lint Adapter

File: `src/linters/remark-lint-adapter.ts`

Implementation:

1. Import `unified`, `remarkParse`, `remarkLint` (or the preset `remark-preset-lint-recommended`).
2. In the `lint()` method, create a unified processor: `unified().use(remarkParse).use(remarkPresetLintRecommended)`.
3. Call `.processSync(source)` to get a `VFile`.
4. Map `VFile.messages` to `LintDiagnostic[]`:
   - Each message has `line`, `column` (1-indexed), `ruleId`, `reason`, `fatal` (boolean or null).
   - Convert `line`/`column` to character offset using the same line-length summation approach.
   - Map `fatal === true` to severity `"error"`, otherwise `"warning"`.
   - Set `ruleId` to `message.ruleId` or `message.source`, `message` to `message.reason`.
5. Wrap in try-catch. On error, return `[]`.

**Note:** remark-lint uses ESM-only packages. The import strategy must match how the MVP codebase handles ESM dependencies. Check whether the existing remark engine (`src/engines/remark-engine.ts`) uses dynamic `import()` or static imports, and follow the same pattern.

#### 6.1 Tighten Tauri CSP

File: `src-tauri/tauri.conf.json`

Replace `"security": { "csp": null }` with:

```json
"security": {
  "csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' https:; font-src 'self' data:;"
}
```

Rationale for each directive:
- `default-src 'self'`: restrict all resources to same-origin by default.
- `script-src 'self'`: only load scripts from the app bundle.
- `style-src 'self' 'unsafe-inline'`: allow bundled styles and inline styles (Tailwind and CodeMirror inject inline styles).
- `img-src 'self' data: https:`: allow local images, data URIs (for SVG embedding), and remote images in rendered markdown content.
- `connect-src 'self' https:`: allow remote URL fetching (the remote markdown fetch feature).
- `font-src 'self' data:`: allow bundled fonts and data URI fonts.

After applying, run the dev server (`pnpm tauri dev` if the Rust toolchain is available, or `pnpm run dev` for web-only) and verify the app loads without CSP violation errors in the browser console.
