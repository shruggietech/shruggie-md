## 1. Executive Summary

All five phases of the v0.3.0 master sprint plan have been implemented in a single atomic commit on `main`. The work delivers a storage architecture migration (JSON → SQLite/IndexedDB), a multi-workspace system replacing the former Library mode, a restructured three-mode view system with collapsible quick-settings toolbar, unified Open and Export dialogs replacing scattered file entry and conversion controls, and a batch of UI polish fixes.

### What Was Delivered

1. **Phase 1 — UI Polish:** Themed scrollbars, reactive editor settings (line numbers, word wrap via Compartment-based reconfig), `:focus-visible` focus rings, button labels toggle, tooltip audit.
2. **Phase 2 — View Mode Restructure:** Three-mode system (View, Edit, Edit Only), five-button segmented toolbar switcher, collapsible EditorToolbarPanel with context-sensitive quick settings.
3. **Phase 3 — Storage Architecture:** `StorageAdapter` interface with SQLite (desktop) and IndexedDB (web/PWA/Chrome) backends; schema for config, documents, workspaces, edit history, logs, and tag stubs; automatic config migration from legacy JSON; structured logging with configurable verbosity; desktop window state persistence.
4. **Phase 4 — Workspace System:** Complete Library → Workspaces terminology migration across all source, tests, config keys, and spec; default workspace with auto-creation and deletion protection; workspace CRUD with internal/external types; per-workspace settings; `PlatformAdapter` extensions (`getAppDataDir`, `createDirectory`, `removeDirectory`).
5. **Phase 5 — I/O Overhaul:** Unified Open dialog (File/URL tabs, workspace selector with inline creation, name override); Export dialog (HTML/PDF/Markdown format selection); New Document button; document model integration (storage registration on open, edit history on save); toolbar action consolidation; `Ctrl+Shift+E` keyboard shortcut.

## 3. Architecture Decisions

### 3.1 Storage Layer Design

The `StorageAdapter` interface provides a unified API for both SQLite and IndexedDB. The factory function `getStorage()` in `src/storage/index.ts` detects `__TAURI__` on `window` to select the backend. Both implementations are lazy-loaded via dynamic `import()` to avoid bundling both backends in every target.

The decision to use a single `settings` JSON column on `workspaces` (rather than a separate `workspace_settings` join table) was made for simplicity — per-workspace settings are always read/written as a unit and the schema is small.

### 3.2 Config Flattening

Config values are stored as dotted-path key-value pairs (`appearance.colorMode` = `"dark"`). The `flattenConfig` / `unflattenConfig` utilities in `config-utils.ts` handle conversion between the nested `AppConfig` type and flat storage representation. This avoids schema changes when config keys are added/removed.

### 3.3 Document Model Integration

Rather than building a full document-centric architecture (which would require significant refactoring of the content state in `App.tsx`), the implementation takes a pragmatic approach: documents are registered in storage at open-time and updated at save-time, but the in-memory state (`content`, `fileName`, `filePath`) remains as simple React state. A `currentDocumentId` tracks the active document for storage operations. This provides the data foundation for future features (recent files, cross-device sync) without over-engineering the current UI.

### 3.4 Open Dialog vs. Direct Open

CLI and OS-association file opens bypass the dialog and route directly into the default workspace. This preserves the fast-open experience for power users while the dialog provides workspace selection for interactive use.

### 3.5 Export Dialog with Direct Shortcuts

The Export dialog (`Ctrl+Shift+E`) coexists with direct shortcuts (`Ctrl+Shift+H` for HTML, `Ctrl+Shift+P` for PDF). The dialog is the primary UI entry point; direct shortcuts remain for users who know what format they want. This avoids breaking established muscle memory.

### 7.1 Not Verified in This Sprint

| Production NSIS build | Full release build not executed. |
| macOS / Linux builds | Not tested. Cross-platform behavior relies on Tauri abstractions. |
| Chrome Extension sideload test | Not executed. IndexedDB backend tested via unit tests with in-memory mock. |

### 7.2 Implementation Pragmatisms

| Item | Notes |
|------|-------|
| `UrlInput` component orphaned | The `UrlInput` component (`src/components/Toolbar/UrlInput.tsx`) is no longer imported by the Toolbar but has not been deleted. It remains in the codebase as dead code. Recommend deleting in a cleanup pass. |
| `useRemoteFetch` hook orphaned | No longer called from `App.tsx` (URL fetching now handled inside `OpenDialog`). Still exported from `hooks/index.ts`. Recommend removing. |
| File copy into workspace not implemented | The plan specifies that opening a file outside a workspace should copy the file into the workspace directory. The current implementation opens external files in-place and records the external path. This is simpler and avoids unexpected file duplication. The document model still records the file in the selected workspace. |
| Save As workspace association | The plan specifies updating workspace association when Save As targets a different workspace directory. The current implementation retains the original workspace association regardless of save location. |
| Edit history pruning | The schema and `pruneEditHistory` method exist, but automatic pruning on startup/save is not wired. The append-only behavior is functional. |

### 7.3 Future Work (Per Plan §13 and Stub Tables)

- **Tags system:** `tags` and `document_tags` tables exist as stubs. No UI wired. Intended for a future milestone.
- **Edit history UI:** Snapshots are recorded but no browsing/restore UI exists.
- **Cross-device sync:** Chrome Extension could use `chrome.storage.sync` for a config subset. Not implemented.
- **Authenticated remote fetching:** Only public endpoints supported.
