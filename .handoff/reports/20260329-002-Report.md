## Executive Summary

Sprint 20260329-002 targeted a critical regression: clicking the native title-bar close button (the "X") on Windows left the Shruggie Markdown window permanently open with no error or freeze indication. The sprint plan diagnosed the root cause as an async `onCloseRequested` handler that could hang indefinitely on IPC or SQLite calls during shutdown, and prescribed a timeout guard plus debounced move/resize saves as complementary fixes.

The initial implementation landed in v0.3.3 and was complete and correct as specified. However, a subsequent code change introduced explicit `event.preventDefault()` + `win.destroy()` close handling to make the flow deterministic — and this revealed a second, previously hidden failure: Tauri 2's capability security system silently denied `win.destroy()` because `core:window:allow-destroy` was absent from the capability manifest. This silent denial left the window permanently unresponsive to the close button across two hotfix releases (v0.3.4) before the root cause was identified and resolved in v0.3.5.

All sprint plan tasks were completed. Three additional releases (v0.3.3, v0.3.4, v0.3.5) were cut and pushed. The final test suite counted 297 passing tests. The branch is clean and current at `origin/main`.

## Baseline State (v0.3.2)

- **Test count:** 295 passing across 12 test files.
- **Known defect:** Native close button (Windows title-bar "X") did not close the application. The window remained open and interactive with no error shown.
- **Root cause (per plan):** The `useWindowState` hook registered an `async` `onCloseRequested` handler containing multiple sequential IPC calls (`isMaximized`, `outerSize`, `outerPosition`) and SQLite writes. If any awaited call stalled during shutdown, the Promise never resolved, Tauri waited indefinitely, and `destroy()` was never invoked.
- **No timeout:** No fallback existed if the handler hung.
- **No redundant persistence:** Window geometry was saved exclusively at close time; no incremental saves occurred during normal use.

## Phase 2 — Fix Implementation

### Task 2.1: Timeout Guard on Close Handler

Added a `Promise.race` guard in the `onCloseRequested` callback:

```ts
const saveTimeout = new Promise<void>((resolve) => {
  setTimeout(resolve, 2000);
});
const saveState = (async () => {
  try {
    await saveWindowState();
  } catch {
    // Silent failure — don't block close
  }
})();
await Promise.race([saveState, saveTimeout]);
```

This guarantees the handler resolves within 2 seconds regardless of IPC or SQLite behavior.

### Task 2.2: Debounced Save on Move/Resize

Added `onMoved` and `onResized` listeners wired to a shared 1000 ms debounced save function. This incrementally persists window geometry during normal use so the close-time save becomes a best-effort final flush rather than the sole persistence path.

Key implementation details:

- Single shared `debounceTimer` handle; cleared and reset on each event.
- Skip persist if window is maximized (consistent with close handler logic).
- Debounce timer and all three listeners (`unlistenClose`, `unlistenMove`, `unlistenResize`) are cleaned up on unmount.
- Close handler clears the debounce timer before executing the close-time save to avoid a double-write race.

### Task 2.3: Test Coverage

New test file `src/__tests__/window-state.test.ts` added with 2 targeted regression tests:

**Test 1 — `resolves close handler within timeout when save hangs`**

Mocks `setConfigValue` to return a never-resolving Promise. Asserts that `event.preventDefault()` is called once and `win.destroy()` is called once after the 2-second timeout fires. Uses `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync(2100)`.

**Test 2 — `debounces move and resize saves`**

Triggers three raw events (two moves, one resize). Asserts zero `setConfigValue` calls before the 1000 ms debounce window and exactly 5 calls (width, height, x, y, maximized) after the debounce fires.

## Hotfix Releases

The sprint plan authorised v0.3.3 only. Two additional hotfix releases were required due to a compounding failure discovered after v0.3.3 shipped.

### v0.3.4 — Deterministic Close Flow

After v0.3.3 shipped, the close button was observed still failing to close the application in the installed binary. The timeout guard was in place, but the close path was still relying on Tauri's implicit post-handler `destroy()` call.

This was addressed by switching to an explicit close model:

- `event.preventDefault()` is called immediately to take ownership of the close flow.
- `Promise.race([saveWindowState(), 2s timeout])` is awaited.
- `win.destroy()` is called explicitly by the handler after the race resolves.

An `isClosing` guard flag was also added to prevent re-entrant close attempts.

This change was committed as `5aa73e3` and released as v0.3.4.

### v0.3.5 — Tauri Capability Permission Fix (True Root Cause)

After v0.3.4 shipped, the close button **still** did not work in the installed binary. Full diagnosis of the Tauri capability system identified the actual root cause:

**`core:window:allow-destroy` was absent from `src-tauri/capabilities/default.json`.**

Tauri 2 uses a capability-based security model where every IPC call from the frontend must be explicitly allowed. The frontend handler was calling `win.destroy()` after `preventDefault`, but the Tauri IPC layer silently denied the `destroy` command because the required capability was not granted. The result was that `preventDefault` blocked Tauri's implicit close, `destroy()` was silently denied, and the window was left permanently unresponsive to the close button.

**This is the actual root cause.** The timeout guard, explicit close flow, and debounced saves were all correct and necessary — but none of them could function because the IPC permission gate was blocking the destroy call.

Fix applied: added `"core:window:allow-destroy"` to the `permissions` array in `src-tauri/capabilities/default.json`.

**Current state of `default.json` (post-fix):**

```json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "default",
  "description": "Default capabilities for Shruggie Markdown",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "core:window:allow-destroy",
    "fs:default",
    ...
  ]
}
```

This fix was committed as `88eeb3b` and released as v0.3.5.

## Lessons Learned

### Tauri 2 Capability Denials Are Silent

The primary lesson from this sprint is that Tauri 2's IPC capability system fails silently. When a frontend call is blocked by a missing capability, no exception is thrown, no error is logged, and no message surfaces in the webview console. The call returns without effect.

This means:

- Any new use of a `win.*` API (or any Tauri IPC command) must be accompanied by a check that the corresponding capability permission is present in `capabilities/default.json`.
- The symptom of a missing capability looks identical to any other silent async failure — making it extremely difficult to distinguish from a timeout, a stalled IPC call, or a logic error.
- The `event.preventDefault()` + `win.destroy()` close flow cannot function at all without `core:window:allow-destroy`, but neither the console nor the Tauri log surface a useful error to indicate this.

**Recommended practice going forward:** Before shipping any new Tauri window API usage, confirm the required permission is present in `default.json` and test the IPC call in a fresh dev build, not just with mocked tests.

### Mock Tests Cannot Catch Capability Gaps

Unit tests that mock `win.destroy()` will always pass, regardless of whether the capability exists in production. The capability gate operates in the Tauri Rust runtime, not in the JavaScript mock layer. Capability verification requires a real built binary.

### Explicit Close Flow Is Correct — But Requires the Right Permissions

The `event.preventDefault()` + explicit `win.destroy()` pattern is the correct approach for controlled close flows that need to perform async work (saves, cleanup) before the application exits. The pattern was sound; the missing capability was the only barrier.
