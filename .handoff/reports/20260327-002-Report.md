## 1. Original Problem

The v0.3.0 milestone code was committed and verified correct at the source level, but the release binaries produced by the build agent contained **stale v0.2.0 frontend output** embedded in a v0.3.0-versioned Tauri shell. The agent had invoked `pnpm tauri build` without first deleting and regenerating `dist/`. Tauri embeds the contents of `dist/` at compile time; because an old `dist/` remained on disk from a prior build, the compiled binary contained the previous frontend verbatim.

This was the third documented instance of an agent shipping unverified or stale artifacts:
- v0.1.0: Agent removed `bundle.windows` from `tauri.conf.json` and never smoke-tested the installed binary.
- v0.1.1: Hotfix worked in dev mode; production NSIS build never executed by the agent.
- v0.3.0: Real source changes were committed but binaries were built against cached frontend output.

---

## 2. Root Cause

Tauri's build architecture is split: the frontend (React/Vite, outputs to `dist/`) and the Rust backend compile independently. The Tauri CLI configuration includes `beforeBuildCommand: "pnpm build"` and `beforeDevCommand: "pnpm dev"`, which should regenerate `dist/` automatically when invoked through the CLI. However, this guard only functions when the Tauri CLI is the entry point.

Two failure paths bypassed it:

1. **Direct Rust invocation.** Running `cargo build` from `src-tauri/` bypasses the Tauri CLI entirely; `beforeBuildCommand` never executes.
2. **Silent frontend build failure.** If the frontend build step fails (e.g., TypeScript error) but a `dist/` directory already exists from a previous successful build, Tauri continues and packages the stale output without error.

Additionally, agents had no automated mechanism enforcing dist cleanup — the protection existed only as a governance rule in `CLAUDE.md`, which agents could overlook or misinterpret.

---

## 3. What Was Done in Response (at the time of 20260327-002)

The remedial plan `20260327-002-Updates.md` addressed the immediate problem and a subset of the systemic gaps:

- **Governance rules added to `CLAUDE.md`:** "Clean frontend build before Tauri build" rule added to Standing Rules §2.7, and the complete Build Procedure (§3) was added with explicit `rm -rf dist` / `Remove-Item` steps before `pnpm run build` and `pnpm tauri build`.
- **Governance rules added to `.github/copilot-instructions.md`:** "Common Pitfalls" section updated with the stale-dist warning.
- **Manual Windows build verification:** William Thompson performed a clean-room build and visual smoke test of the v0.3.0 desktop installer on Windows, confirming the binaries were correct. Documented in `20260327-001-Report.md` §10.

---

## 4. What Was NOT Done at the Time

The following items were identified in the audit but deferred:

- **No automation guardrails in `package.json`.** The `tauri:dev` and `tauri:build` scripts still invoked the Tauri CLI directly (`"tauri dev"`, `"tauri build"`) with no dist cleanup step. An agent or contributor who ran these scripts directly — without consulting `CLAUDE.md` — would still encounter the stale-dist failure mode.
- **No contributor-facing documentation in README.md.** The build hygiene requirement was recorded only in AI agent governance files, not in the public contributor setup guide that human contributors would follow.
- **No release process documentation in RELEASING.md.** The release checklist did not include a clean-build verification step before tagging.

The remaining gaps are addressed by sprint plan `20260328-001-Updates.md` (executed 2026-03-28):

---

## 6. Status

**Closed.** The stale-dist problem is now defended at three layers:

1. **Automation** — `package.json` scripts delete `dist/` before every Tauri invocation.
2. **Documentation** — `README.md` and `RELEASING.md` make the requirement visible to human contributors and release managers.
3. **Governance** — `CLAUDE.md` reinforces the requirement for AI agents and specifies the correct npm script entry points.

No further action required under plan 20260327-002.
