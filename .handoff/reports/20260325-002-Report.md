## 3. CI/CD Failure Analysis & Resolution

The release workflow required three attempts before full success. This section documents each failure, its root cause, the investigation method, and the fix applied.

### Attempt 1 — pnpm Version Conflict

**Commit:** `2c16d54 feat: Shruggie Markdown v0.1.0 MVP`
**Run ID:** 23529065026 (Release), 23529062559 (CI)
**Result:** CI failed, Release failed — all jobs failed at "Setup pnpm" step

**Error:**
```
Error: Multiple versions of pnpm specified:
  - version 9 in the GitHub Action config with the key "version"
  - version pnpm@10.28.2 in the package.json with the key "packageManager"
```

**Root Cause:** The sprint plan template specified `version: 9` for `pnpm/action-setup@v4`, but `package.json` was created by `pnpm init` which set `"packageManager": "pnpm@10.28.2"`. The action setup v4 detects both and refuses to proceed when they disagree.

**Investigation:** Read the failed step logs via `gh run view --log-failed`.

**Fix (commit `0b547c7`):** Removed all 4 occurrences of `version: 9` from `pnpm/action-setup@v4` across `ci.yml` and `release.yml`. The action now auto-detects from the `packageManager` field.

**Cleanup:** Deleted draft release, deleted remote+local tag, re-tagged on fix commit.

### Attempt 2 — Tauri Config Schema Validation

**Commit:** `0b547c7 fix: remove explicit pnpm version from CI/CD workflows`
**Run ID:** 23529455892 (Release), 23529455410 (CI)
**Result:** CI passed (success). Release partially failed — create-release and build-extension succeeded; all 3 build-tauri jobs failed; publish-release skipped.

**Error:**
```
Error `"tauri.conf.json"` error on `bundle > windows > wix`: {"fileAssociations":[...]} is not valid under any of the schemas listed in the 'anyOf' keyword
Error `"tauri.conf.json"` error on `bundle > macOS`: Additional properties are not allowed ('fileAssociations' was unexpected)
Error `"tauri.conf.json"` error on `bundle > linux`: Additional properties are not allowed ('desktop' was unexpected)
```

**Root Cause:** The Sprint 10 sub-agent (from the MVP build) placed file associations and Linux desktop configuration in the wrong schema locations. In Tauri 2:
- `fileAssociations` is a **top-level** property of `BundleConfig`, not nested inside `windows.wix`, `macOS`, or `linux`
- `FileAssociation.ext` is an **array of strings**, not a single string
- Linux `LinuxConfig` has `appimage`, `deb`, `rpm` — no `desktop` or `categories` properties
- macOS `MacConfig` has `frameworks`, `dmg`, `signingIdentity`, etc. — no `fileAssociations`
- WiX `WixConfig` has `version`, `upgradeCode`, `language`, etc. — no `fileAssociations`

**Investigation:** Programmatically inspected the Tauri 2 JSON schema at `node_modules/@tauri-apps/cli/config.schema.json` using Node.js to enumerate valid properties for `BundleConfig`, `WindowsConfig`, `WixConfig`, `LinuxConfig`, `MacConfig`, and `FileAssociation`.

**Fix (commit `ecbcf6b`):**
- Moved `fileAssociations` to `bundle.fileAssociations` with `ext` as an array
- Removed the entire `bundle.windows` block (WiX sub-config was unnecessary)
- Removed `bundle.macOS` block entirely
- Replaced `bundle.linux.desktop` with valid `bundle.linux.deb.section`
- Added `bundle.license` and `bundle.category` at the correct level

**Cleanup:** Deleted draft release, deleted remote+local tag, re-tagged on fix commit.

### Attempt 3 — PNG Color Type (Non-RGBA Icons)

**Commit:** `ecbcf6b fix: correct tauri.conf.json bundle schema for Tauri 2`
**Run ID:** 23530062604 (Release), 23530062094 (CI)
**Result:** CI passed. Release partially succeeded — create-release, build-extension, and **build-tauri (windows-latest)** all succeeded. build-tauri on ubuntu-22.04 and macos-latest failed. publish-release skipped.

**Error:**
```
error: proc macro panicked
  --> src/lib.rs:50:14
   |
50 |         .run(tauri::generate_context!())
   |              ^^^^^^^^^^^^^^^^^^^^^^^^^^
   |
   = help: message: icon /home/runner/work/shruggie-md/shruggie-md/src-tauri/icons/32x32.png is not RGBA
```

**Root Cause:** The master icon `assets/brand/icon-master.png` is a palette-indexed PNG (only 10 colors). When ImageMagick resized it, it preserved the palette color mode, producing PNG files with color type 3 (indexed). Tauri's `generate_context!()` proc macro requires RGBA (color type 6) PNGs. Windows succeeded because the Windows bundler path doesn't validate icon color types through the same proc macro path.

**Investigation:**
1. Identified the exact error message via `gh run view --log --job <id> | grep "proc macro panicked" -B5 -A10`
2. Confirmed the PNG color type locally: `magick identify -format '%[png:IHDR.color-type-orig]' src-tauri/icons/32x32.png` returned `3` (indexed)
3. Determined the fix: use ImageMagick's `PNG32:` output prefix to force color type 6 (RGBA truecolor with alpha)

**Fix (commit `ad61833`):**
- Regenerated all 15 Tauri PNG icons using `magick <input> -resize <size> -alpha on PNG32:<output>`
- Verified all files are now color type 6 via `magick identify -format '%[png:IHDR.color-type-orig]'`
- Updated both `generate-icons.sh` and `generate-icons.ps1` to use `PNG32:` prefix for future regeneration

**Cleanup:** Deleted draft release, deleted remote+local tag, re-tagged on fix commit.
